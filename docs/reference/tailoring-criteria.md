# プロジェクト特性ヒアリング項目（C1-C10）

> このファイルは `/tailoring` の参照資料です。
> CLAUDE.md から `@` 参照でオンデマンド読み込みされます。

---

`/tailoring` でプロジェクト特性を把握するための質問項目。
各特性が、後述のプロセスゴールの推薦判断に影響する。

### C1. 契約形態

| 選択肢 | 説明 |
|--------|------|
| **準委任** | 工数ベース。スコープの柔軟性が高い |
| **請負** | 成果物ベース。スコープ合意と変更管理が重要 |
| **社内プロジェクト** | 社内ステークホルダーとの調整が中心 |

**影響するプロセスゴール**: リリース計画、見積もり、ステークホルダー連携、ドキュメント、スコープ管理

### C2. 顧客の関与度

| 選択肢 | 説明 |
|--------|------|
| **密（週次以上）** | PO的に深く関与。フィードバックが早い |
| **中（隔週〜月次）** | 定期的な確認はあるが日常的ではない |
| **疎（月次未満）** | 確認の機会が限られる。期待値のズレに注意 |

**影響するプロセスゴール**: ステークホルダー連携、リリース計画、フィードバックループの設計

### C3. チームのアジャイル成熟度

| 選択肢 | 説明 |
|--------|------|
| **初めて** | スクラムの基本から丁寧に。ガードレールを多めに |
| **経験あり** | 基本はわかっている。プラクティスの取捨選択ができる |
| **熟練** | 自律的に動ける。最小限のプロセスで最大の効果を出せる |

**影響するプロセスゴール**: ほぼ全て。成熟度が低いほどプラクティスを明示的に定義する必要がある

### C4. ドメインの不確実性

| 選択肢 | 説明 |
|--------|------|
| **高い（新規ドメイン）** | チームに業務知識が少ない。学びながら進む |
| **中（類似経験あり）** | 似た経験はあるが、このドメイン固有の学びが必要 |
| **低い（熟知している）** | チームがドメインに詳しい。要件の解像度が高い |

**影響するプロセスゴール**: リスク管理、アーキテクチャ、バックログ整備、テスト戦略

### C5. 技術的複雑性

| 選択肢 | 説明 |
|--------|------|
| **高い** | 新技術の採用、複雑なインテグレーション、未知の課題が多い |
| **中** | 一部に新技術や不確実な要素がある |
| **低い** | 既知の技術スタックで、チームに十分な経験がある |

**影響するプロセスゴール**: アーキテクチャ、テスト戦略、リスク管理、DoD

### C6. 品質要求レベル

| 選択肢 | 説明 |
|--------|------|
| **標準** | 一般的なWebアプリケーション水準 |
| **高い** | 金融・医療・製造業の基幹系など。障害の影響が大きい |
| **非常に高い** | 人命・安全に関わる。規制対応必須 |

**影響するプロセスゴール**: テスト戦略、コードレビュー、DoD、品質ゲート、非機能要件

### C7. チーム規模

| 選択肢 | 説明 |
|--------|------|
| **小（3-5名）** | 全員の顔が見える。口頭コミュニケーションで十分なことが多い |
| **中（6-9名）** | スクラムの適正サイズ。明示的なコミュニケーション設計が必要 |
| **大（10名以上）** | 複数チーム or サブチーム。スケーリングの仕組みが必要 |

**影響するプロセスゴール**: デイリーの進め方、ドキュメント、コミュニケーション設計

### C8. チーム分散度

| 選択肢 | 説明 |
|--------|------|
| **同一拠点** | 対面コミュニケーションが容易 |
| **フルリモート** | 意図的な同期の場と非同期コミュニケーションの設計が必要 |
| **ハイブリッド / 複数拠点** | 情報格差が生まれやすい。透明性の仕組みが重要 |

**影響するプロセスゴール**: デイリーの進め方、ドキュメント、ステークホルダー連携

### C9. 規制・コンプライアンス要件

| 選択肢 | 説明 |
|--------|------|
| **なし** | 特段の規制なし |
| **軽い** | 社内セキュリティポリシー準拠、基本的な監査対応 |
| **厳しい** | 業界規制（ISMS、SOC2、医療機器規制等）への準拠が必要 |

**影響するプロセスゴール**: ドキュメント、テスト戦略、品質ゲート、DoD

### C10. プロジェクト期間

| 選択肢 | 説明 |
|--------|------|
| **短期（〜3ヶ月）** | PoC・MVP。スピード重視。最小限のプロセス |
| **中期（3-12ヶ月）** | 本開発。リリース計画とリズムの確立が重要 |
| **長期（1年以上）** | 持続可能性が鍵。チームの成長、技術的負債管理も視野に |

**影響するプロセスゴール**: リリース計画、メトリクス、レトロスペクティブ、リスク管理
