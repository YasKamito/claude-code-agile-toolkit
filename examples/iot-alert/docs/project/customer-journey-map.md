## ジャーニー1: 日勤帯の通常監視（本田さん）

| フェーズ | 現状（As-Is） | あるべき姿（To-Be） |
|---------|--------------|-------------------|
| **朝の確認** | PLC画面を1台ずつ巡回して確認（約30分） | ダッシュボードで全設備を一覧確認（赤黄青シグナル）。1分で異常の有無がわかる |
| **日中の監視** | 設備の音や振動を五感で確認。定期巡回（2時間ごと） | リアルタイムモニタリング。閾値超えは即時Slack通知。巡回間隔を最適化 |
| **異常検知** | 異音や数値異常に気づく → 経験と勘で判断 | 「注意」通知で早期に気づく。トレンドグラフで「いつもと違う」を数値で確認 |
| **対応判断** | 過去の記憶に頼って判断。記録は紙のノート | 過去データの履歴閲覧で根拠に基づく判断。対応履歴もデジタル化（将来） |
| **引き継ぎ** | 口頭 or ノートで夜勤担当に伝える | ダッシュボードの状態が引き継ぎ資料になる。要注意設備が一目瞭然 |

**ペインポイント:**
- PLC画面の巡回に時間がかかる
- 経験の浅い担当者に「いつもと違う」を伝えられない
- 紙の記録は検索できない

## ジャーニー3: 管理者の状況把握（村田さん）

| フェーズ | 現状（As-Is） | あるべき姿（To-Be） |
|---------|--------------|-------------------|
| **日次確認** | 保全チームからの口頭報告。問題がなければ報告なし | ダッシュボードで全設備の稼働状況を俯瞰。異常なしも確認できる |
| **異常発生時** | 電話連絡を受けて現場に駆けつける | Slack通知で即座に状況把握。ダッシュボードで詳細確認 |
| **他拠点の状況** | 岡崎・三重に電話して確認 | 全拠点の状態を1画面で確認（マルチ拠点対応後） |
| **月次レビュー** | 紙の故障記録を集めて集計 | アラート履歴・トレンドデータから自動集計（将来） |

**ペインポイント:**
- 「問題なし」の確認に手間がかかる
- 他拠点の状況がリアルタイムでわからない
- データに基づく予防保全計画が立てられない

## MVP で解決すべき最重要ジャーニー

1. **ジャーニー2（夜勤帯のアラート対応）** — 夜勤1人体制のリスクが最も高い。アラート通知の価値が最大
2. **ジャーニー1（日勤帯の通常監視）** — ダッシュボードによる業務効率化。ベテランの知見をデータに変換
3. **ジャーニー3（管理者の状況把握）** — ダッシュボードの副次的な価値

→ MVP は「閾値アラート通知 + ダッシュボード」でジャーニー1・2をカバーする
