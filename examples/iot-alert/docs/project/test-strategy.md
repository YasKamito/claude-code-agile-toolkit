# テスト戦略 — IoTアラート

> 最終更新: 2026-03-30
> ステータス: ドラフト
> 採用プラクティス: CI組み込みテスト + 探索的テスト

---

## 方針

- **シフトレフト**: テストは開発と同時に作る。後工程で品質を確保しない
- **自動化優先**: 繰り返し実行するテストはCI/CDに組み込む
- **探索的テストで補完**: 自動テストでカバーしにくいUX・業務フローは探索的テストで検証

---

## テストレベル別方針

### 1. ユニットテスト（CI組み込み）
- **対象**: ビジネスロジック（閾値判定、通知レベル振り分け、通知先ルーティング）
- **ツール**: Jest / pytest（技術スタック確定後に決定）
- **カバレッジ目標**: 閾値判定・通知ロジックは80%以上。UI層は対象外
- **実行タイミング**: PRごとに自動実行

### 2. インテグレーションテスト（CI組み込み）
- **対象**: PLCゲートウェイAPI連携、データストア読み書き、通知送信（Slack/メール）
- **方針**: 外部APIはモック化。データストアはテスト用インスタンスを使用
- **実行タイミング**: マージ時に自動実行

### 3. E2Eテスト（手動 + 段階的自動化）
- **対象**: 主要ユーザーフロー（ダッシュボード閲覧、閾値設定変更、アラート受信確認）
- **初期**: 手動テスト + チェックリスト（スプリントレビュー前に実施）
- **中期**: 主要フローのみ自動化を検討

### 4. 探索的テスト（毎スプリント）
- **目的**: カスタマージャーニーマップのペインポイントが解消されているか検証
- **重点領域**:
  - 閾値設定UI（保全担当者が迷わず操作できるか）
  - アラート通知（通知の遅延、見逃し、オオカミ少年化していないか）
  - ダッシュボード（全設備の状態が一目でわかるか）
  - 夜勤帯シナリオ（スマホでSlack通知を受け取り、状況判断できるか）
- **実施者**: 開発チーム（ローテーション）。可能なら本田さんにも触ってもらう
- **記録**: 発見事項をPBIとしてバックログに追加

## 環境

| 環境 | 用途 | データ |
|------|------|--------|
| ローカル | 開発・ユニットテスト | モックデータ |
| ステージング | インテグレーション・E2E・探索的テスト | モックPLC APIからのサンプルデータ |
| 本番 | 本番稼働 | 実PLCゲートウェイからのリアルデータ |

---

## セキュリティテスト（規制対応: C9）

- VPN接続の検証（VPN外からのアクセスが遮断されること）
- 認証・認可の検証（未認証ユーザーのアクセス拒否）
- アクセスログの記録検証（誰がいつ何を見たか）
- APIキーの管理・ローテーション手順の確認

---

## 判断基準

| 判断 | 基準 |
|------|------|
| リリース可 | CI全テストグリーン + 探索的テストで重大な問題なし + PO受入OK |
| リリース不可 | 閾値判定の不具合、通知未達、セキュリティ要件未充足 |
