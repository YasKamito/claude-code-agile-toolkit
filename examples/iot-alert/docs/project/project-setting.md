## プロジェクト特性（/tailoring ヒアリング結果）

| 特性 | 回答 | 備考 |
|------|------|------|
| C1. 契約形態 | 準委任 | スコープの柔軟性が高い |
| C2. 顧客の関与度 | 密（週次以上） | POとして深く関与。村田さん・本田さんから具体的な要望あり |
| C3. アジャイル成熟度 | 初めて | 田中は経験あり、他メンバーは初めて〜浅い。全体としては「初めて」寄り。ガードレールを多めに |
| C4. ドメインの不確実性 | 高い（新規ドメイン） | 製造業IoT・設備監視はチームにとって新領域 |
| C5. 技術的複雑性 | 中 | PLCゲートウェイ連携、マルチ拠点対応に不確実性あり |
| C6. 品質要求レベル | 標準 | 一般的なWebアプリケーション水準 |
| C7. チーム規模 | 小（5名） | PO 1 + SM 1 + Dev 3 |
| C8. チーム分散度 | ハイブリッド / 複数拠点 | 情報格差が生まれやすい。透明性の仕組みが重要 |
| C9. 規制・コンプライアンス要件 | 厳しい | VPN必須、アクセスログ必須、AD連携。業界規制への準拠が必要 |
| C10. プロジェクト期間 | 短期（〜3ヶ月） | スピード重視。MVP先行 |

## 採用プラクティス

### 立ち上げ（Inception）
- **プロダクトの方向性**: インセプションデッキ + カスタマージャーニーマップ
- **ステークホルダー整理**: ステークホルダーマップ
- **初期リスク**: リスクバックログ + プレモーテム
- **働き方**: ワーキングアグリーメント

### 計画（Planning）
- **リリース計画**: スプリントごとリリース
- **バックログ整備**: INVEST基準 + DoR（Definition of Ready）定義 + ユーザーストーリーマッピング
- **見積もり**: 相対見積もり（SP）のみ
- **テスト戦略**: CI組み込みテスト + 探索的テスト
- **アーキテクチャ方針**: スパイクで検証 + ADR
- **DoD**: コードレビュー必須 + セキュリティレビュー

### 実行（Execution）
- **デイリー**: 毎日15分同期
- **障害管理**: PBLに混ぜて対応
- **ステークホルダー連携**: 毎スプリントレビュー
- **リスク継続管理**: リスクバックログを定期レビュー
- **ドキュメント管理**: コンプライアンス対応

### 品質（Quality）
- **コードレビュー**: PRレビュー必須
- **テスト実行**: CI/CDに組み込み + 探索的テスト併用
- **非機能要件**: PBLに混ぜて対応

### ふりかえり・改善（Review & Retrospective）
- **スプリントレビュー**: ＋ステークホルダー参加
- **レトロスペクティブ**: 毎スプリント
- **メトリクス**: ベロシティのみ

## Playbook

### Sprint 0（立ち上げスプリント）でやること
- [ ] インセプションデッキ作成（`/inception-deck`）
- [ ] カスタマージャーニーマップ作成（保全担当者・夜勤帯・マルチ拠点の業務フロー）
- [ ] ステークホルダーマップ作成（3拠点の関係者整理）
- [ ] リスクバックログ初期化 + プレモーテム実施
- [ ] ワーキングアグリーメント策定（ハイブリッド勤務の同期・非同期ルール）
- [ ] DoD定義（コードレビュー必須 + セキュリティレビュー）
- [ ] テスト戦略策定（CI組み込み + 探索的テスト） ⚠️ スキル未実装（手動対応）
- [ ] リリース計画策定（スプリントごとリリース方針） ⚠️ スキル未実装（手動対応）
- [ ] アーキテクチャスパイク（PLCゲートウェイAPI連携の技術検証）
- [ ] ADR運用開始（アーキテクチャ判断の記録）
- [ ] DoR（Definition of Ready）定義（PBIが「着手可能」と言える条件を明文化）
- [ ] バックログ初期整備（INVEST基準 + ユーザーストーリーマッピング）
- [ ] スケジュール策定（`docs/project/schedule.md` にスプリント日程と会議体を記録）
- [ ] コンプライアンス対応ドキュメントの対象範囲決定（VPN設計、アクセスログ方針、AD連携設計等）

### 毎スプリントでやること
- [ ] スプリントプランニング（`/planning`）
  - PBI品質確認（`/refinement check` → `/refinement ac`）
  - スプリントゴール設定
- [ ] デイリー（`/daily`）
  - 毎日15分同期
  - ブロッカー検出・対応
- [ ] リスクバックログレビュー（プランニングまたはレトロで）
- [ ] スプリントレビュー（`/sprint-review`）
  - ステークホルダー参加でデモ
  - 村田さん・本田さんからの直接フィードバック収集
- [ ] レトロスペクティブ（`/retro`）
  - 前回アクションのフォロー
  - ベロシティ確認
- [ ] 探索的テスト実施（閾値設定UI、アラート通知の使い勝手検証）
- [ ] コンプライアンス関連ドキュメントの更新（必要に応じて）
