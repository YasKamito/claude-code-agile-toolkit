# ADR-001: PLCゲートウェイAPI連携方式

- **ステータス**: 提案中
- **日付**: 2026-03-30
- **決定者**: 田中 亮

---

## コンテキスト

東海ミスリル工業の工場設備（プレス機、溶接ロボット、塗装ライン、検査装置）のセンサーデータを取得するため、PLCベンダー（アダマンタイト・オートメーション）が提供するPLCゲートウェイのREST APIを利用する。

- API仕様は2026年4月第2週に確定予定（現在ドラフト版）
- データ取得方式: ポーリング（最短5秒間隔）
- 認証: APIキーベース
- 本社工場のみ先行。岡崎・三重は5月以降

## 決定すべきこと

1. ポーリング間隔をどうするか（5秒〜60秒）
2. データ取得のアーキテクチャ（直接ポーリング vs メッセージキュー経由）
3. API仕様未確定期間のモック戦略

## 選択肢

### A. Lambda + EventBridge による定期ポーリング
- EventBridgeで定期実行 → Lambda → API呼び出し → DynamoDB/TimeStream
- シンプル。AWS IoT Coreを使わない軽量構成

### B. AWS IoT Core + ルールエンジン
- PLCゲートウェイ → AWS IoT Core（MQTT） → ルールエンジン → DynamoDB/TimeStream
- ただし現状のAPI仕様はREST。MQTT変換が必要

### C. ハイブリッド（Lambda ポーリング → IoT Core）
- Lambdaでポーリングし、IoT Coreに流す
- 将来的なMQTT対応への拡張性を確保

## 決定

**Sprint 0のスパイクで検証して決定する。**

ドラフト版APIでA案（Lambda + EventBridge）のプロトタイプを構築し、
データ取得の安定性・レイテンシを評価する。

## 影響

- API仕様確定までモックAPIで開発を進める
- スパイクの結果はADR-001を更新して記録する
