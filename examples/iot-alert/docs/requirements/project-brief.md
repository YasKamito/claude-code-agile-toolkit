## 背景・課題

- 設備故障が起きてからの事後対応が常態化。予防保全に移行したい
- ベテラン保全担当者の経験と勘に依存（「音で8割わかる」）。世代交代が進む中、属人性が課題
- 夜勤帯は保全1名体制。異常検知が遅れると設備損傷が拡大するリスク
- 3拠点間の状況確認が電話ベース。リアルタイムの可視化がない
- PLCにセンサーデータは蓄積されているが、活用できていない（PLC画面で確認するのみ）

## 制約・前提

- PLCゲートウェイはベンダー（アダマンタイト・オートメーション）が構築。REST APIでデータ提供
- API仕様は2026年4月第2週に確定予定（現在ドラフト版）
- 岡崎・三重工場のゲートウェイは2026年5月以降設置予定
- 通信断の場合、ゲートウェイ側で最大30分のデータをバッファリング。復旧後に一括送信
- クラウドへの接続はVPN必須（大野さん要件）
- サーバー運用はCM側で実施
- PLCの種類により取得可能なセンサー項目が異なる（対応表は別途共有予定）

## 通知要件（ヒアリングより）

### 通知手段
- Slack（社内導入済み）+ メール（Slack見ない人用）
- 電話通知は将来対応

### 通知レベル
| レベル | 条件 | 通知手段 |
|--------|------|---------|
| 注意（黄色） | 閾値の80%超え | Slackのみ |
| 警告（赤） | 閾値超え | Slack + メール |
| 異常（緊急） | 閾値の120%超え or センサー断 | 全員に通知 |

### 通知先グループ
- 拠点別 × 設備種別（例: 本社プレス担当、岡崎全般担当）

## PLC API情報（ドラフト版）

- エンドポイント: `GET /api/v1/devices/{device_id}/sensors`
- データフォーマット: JSON
- 取得頻度: 最短5秒間隔でポーリング可能
- 認証: APIキーベース

```json
{
  "device_id": "PRESS-001",
  "device_name": "大型プレス1号機",
  "factory": "honsha",
  "timestamp": "2026-04-01T10:00:00+09:00",
  "sensors": [
    { "sensor_id": "temp_mold", "name": "金型温度", "value": 58.3, "unit": "℃" },
    { "sensor_id": "vibration", "name": "振動", "value": 2.1, "unit": "mm/s" },
    { "sensor_id": "pressure", "name": "油圧", "value": 15.2, "unit": "MPa" }
  ],
  "status": "online"
}
```
