## ヒアリング内容（14:40-16:00）

### 村田さんの要望
1. **とにかくまず通知が欲しい**。壊れてから知るのではなく、壊れる前に知りたい
2. ダッシュボードは「全設備の状態が一目でわかる画面」がほしい。赤黄青みたいなシグナル
3. **岡崎と三重の担当者が、本社の状況も見られるようにしたい**（今は電話で確認している）
4. 通知先は担当者ごとに分けたい。プレス担当にはプレスのアラート、溶接担当には溶接のアラートだけ
5. 将来的にはAIで予兆検知したいが、「まずは閾値で十分。ちゃんと動くものを見せてくれ」

### 本田さんの懸念
- 「閾値は設備ごとに全部違う。誰がそれを設定するの？」
- 「通知が多すぎると意味がない。オオカミ少年になる。重要度のレベル分けは必要」
- 「過去データも見たい。いつ頃から数値がおかしくなっていたか、振り返りたい」
- 「PLCの通信が止まったらどうするの？通信異常自体もアラートにしてほしい」

### 大野さん（IT）の発言
- 「クラウドに出すならVPNは必須。直接インターネットに出すのは許可できない」
- 「ユーザー管理はActive Directory連携してほしい。別々にID管理したくない」
- 「アクセスログは必須。誰がいつ何を見たか残してほしい」
- 「サーバーの運用は御社（CM）でやってほしい。うちには人がいない」

### 通知に関する詳細
- **通知手段**: Slack（社内導入済み）+ メール（Slack見ない人用）
- **通知レベルの要望**:
  - 注意（黄色）: 閾値の80%超え → Slackのみ
  - 警告（赤）: 閾値超え → Slack + メール + 担当者に電話（電話は将来）
  - 異常（緊急）: 閾値の120%超え or センサー断 → 全員に通知
- **通知先グループ**: 拠点別 × 設備種別 で分けたい（例: 本社プレス担当、岡崎全般担当）

### 運用面
- 閾値の設定・変更は**保全担当者が自分でできる**ようにしたい（IT部門に依頼する運用は回らない）
- 設備の追加・削除も簡単にしたい（年に数回、設備の入れ替えがある）
- **夜勤帯の通知が特に重要**（夜勤は保全1名体制）

## PLCベンダー（アダマンタイト・オートメーション）からの情報

### データ取得方式
- PLCゲートウェイを設置し、REST APIでデータを提供する（ベンダー側で構築中）
- エンドポイント: `GET /api/v1/devices/{device_id}/sensors`
- データフォーマット: JSON
- 取得頻度: **最短5秒間隔**でポーリング可能
- 認証: APIキーベース

### 提供データ項目（予定）
```json
{
  "device_id": "PRESS-001",
  "device_name": "大型プレス1号機",
  "factory": "honsha",
  "timestamp": "2026-04-01T10:00:00+09:00",
  "sensors": [
    { "sensor_id": "temp_mold", "name": "金型温度", "value": 58.3, "unit": "℃" },
    { "sensor_id": "vibration", "name": "振動", "value": 2.1, "unit": "mm/s" },
    { "sensor_id": "pressure", "name": "油圧", "value": 15.2, "unit": "MPa" }
  ],
  "status": "online"
}
```

### 注意事項
- API仕様は4月第2週に確定予定（現在ドラフト版）
- 岡崎・三重工場のゲートウェイは5月以降の設置予定
- **通信断の場合、最大30分のデータをゲートウェイ側でバッファリング**。復旧後に一括送信
- PLCの種類により取得可能なセンサー項目が異なる（対応表は別途共有予定）
