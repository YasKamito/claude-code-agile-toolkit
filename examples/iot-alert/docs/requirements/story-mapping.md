# ユーザーストーリーマッピング — IoTアラート

> 最終更新: 2026-03-30

---

## ユーザー種別

| ユーザー | IDプレフィックス | 説明 | 主な関心事 |
|----------|----------------|------|-----------|
| 保全担当者 | S- | 日勤・夜勤で設備を監視・保全する | アラートで異常に早く気づきたい。判断材料がほしい |
| 管理者 | M- | 工場管理部門長。全体を俯瞰 | 全設備の状態を一目で把握したい |
| システム管理者 | A- | IT部門。システム設定・ユーザー管理 | セキュリティ・運用を安全に回したい |

---

## ストーリーマップ

### ナラティブフロー

```mermaid
flowchart LR
    subgraph 保全担当者
        S1[ダッシュボードを見る] --> S2[アラートを受け取る]
        S2 --> S3[詳細を確認する]
        S3 --> S4[対応を判断する]
        S4 --> S5[過去データを振り返る]
    end
    subgraph 管理者
        M1[全体状況を把握する] --> M2[拠点別に確認する]
    end
    subgraph 設定・管理
        A1[閾値を設定する] --> A2[通知先を管理する]
        A2 --> A3[ユーザーを管理する]
    end
```

### 保全担当者（S-）

| アクティビティ | ID | ストーリー | 優先度 | リリース |
|---------------|-----|-----------|--------|---------|
| ダッシュボードを見る | S-001 | 保全担当者として、全設備の状態を一覧で確認したい。なぜなら巡回せずに異常の有無を即座に把握したいから | 高 | MVP |
| | S-002 | 保全担当者として、設備の状態を赤黄青のシグナルで見たい。なぜなら一目で正常/注意/異常がわかるから | 高 | MVP |
| | S-003 | 保全担当者として、設備をタップして詳細画面に遷移したい。なぜなら気になる設備の情報をすぐ確認したいから | 高 | MVP |
| アラートを受け取る | S-010 | 保全担当者として、閾値超えのアラートをSlackで受け取りたい。なぜなら巡回していなくても異常に気づけるから | 高 | MVP |
| | S-011 | 保全担当者として、警告以上のアラートをメールでも受け取りたい。なぜならSlackを見逃しても気づけるようにしたいから | 高 | MVP |
| | S-012 | 保全担当者として、自分の担当設備のアラートだけ受け取りたい。なぜなら関係ないアラートが多いと見なくなるから | 高 | MVP |
| | S-013 | 保全担当者として、通信異常（センサー断）のアラートを受け取りたい。なぜならデータが来ないこと自体が危険信号だから | 高 | MVP |
| 詳細を確認する | S-020 | 保全担当者として、設備のセンサー値をリアルタイムで確認したい。なぜなら現在の状態を正確に把握したいから | 高 | MVP |
| | S-021 | 保全担当者として、センサー値のトレンドグラフを見たい。なぜなら「いつもと違う」を数値の推移で判断したいから | 高 | MVP |
| | S-022 | 保全担当者として、閾値のラインをグラフ上に表示したい。なぜなら現在値と閾値の関係が直感的にわかるから | 中 | v1 |
| 対応を判断する | S-030 | 保全担当者として、アラートの通知レベル（注意/警告/異常）を確認したい。なぜなら緊急度に応じた対応をしたいから | 高 | MVP |
| 過去データを振り返る | S-040 | 保全担当者として、過去のセンサーデータ推移を閲覧したい。なぜなら異常がいつ頃から始まっていたか振り返りたいから | 中 | v1 |
| | S-041 | 保全担当者として、過去のアラート履歴を検索したい。なぜなら同じ設備で繰り返し発生している問題を把握したいから | 中 | v1 |

### 管理者（M-）

| アクティビティ | ID | ストーリー | 優先度 | リリース |
|---------------|-----|-----------|--------|---------|
| 全体状況を把握する | M-001 | 管理者として、全設備の稼働状況を俯瞰的に確認したい。なぜなら生産ラインの健全性を把握したいから | 高 | MVP |
| 拠点別に確認する | M-010 | 管理者として、拠点ごとに設備状況をフィルタしたい。なぜなら岡崎・三重の状況も一画面で確認したいから | 中 | v1 |

### システム管理者 / 設定（A-）

| アクティビティ | ID | ストーリー | 優先度 | リリース |
|---------------|-----|-----------|--------|---------|
| 閾値を設定する | A-001 | 保全担当者として、設備ごとにセンサーの閾値を設定・変更したい。なぜなら設備ごとに基準が違い、自分で調整する必要があるから | 高 | MVP |
| | A-002 | 保全担当者として、閾値の変更履歴を確認したい。なぜなら誰がいつ変えたか追跡したいから | 低 | v1 |
| 通知先を管理する | A-010 | 管理者として、通知先グループ（拠点別×設備種別）を設定したい。なぜなら担当者に適切なアラートだけ届けたいから | 高 | MVP |
| ユーザーを管理する | A-020 | システム管理者として、ユーザーの追加・削除・権限設定をしたい。なぜならアクセス管理が必要だから | 中 | MVP |
| | A-021 | システム管理者として、AD連携でユーザー認証したい。なぜなら別々にID管理したくないから | 中 | v2 |
| | A-022 | システム管理者として、アクセスログを確認したい。なぜなら誰がいつ何を見たか記録が必要だから | 中 | v1 |

## リリース計画（概要）

- **MVP（Sprint 1-4 / 4月末）**: 本社工場17台の監視・アラート通知・ダッシュボード最小版。保全担当者が「壊れる前に気づける」状態
- **v1（Sprint 5-8 / 5月末）**: 履歴閲覧、閾値設定UI強化、マルチ拠点対応、アクセスログ
- **v2（Sprint 9-12 / 6月末）**: AD連携、監査レポート、運用最適化
