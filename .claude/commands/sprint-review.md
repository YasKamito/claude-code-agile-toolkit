# /sprint-review - スプリントレビュー支援

スプリントレビューの準備（デモシナリオ作成）と、レビュー後のフィードバック整理を支援する。
POやステークホルダーに「動くソフトウェア」を見せて、フィードバックを得るための場をサポートする。

結果は `docs/sprints/sprint-N/review.md` に保存する。

---

## ファイル構成

```
docs/sprints/sprint-N/
├── planning.md         # /planning の出力
├── daily-log.md        # /daily の出力
├── review.md           # ← このスキルの出力
└── retrospective.md    # /retro の出力
```

---

## 実行フロー

### Step 1: 現在の状態を確認する

以下の情報を収集する:

1. このスキルファイルの手順に従って進める（CLAUDE.mdの全文読み込みは不要。プロジェクト情報は project-setting.md から取得する）
2. **`docs/project/project-setting.md`** を読み、プロジェクト情報・採用プラクティス・Playbookを把握する
3. `docs/sprints/sprint-N/planning.md` を読み、今スプリントの計画を把握する
   - スプリントゴール
   - 選定PBI一覧
4. `docs/sprints/sprint-N/daily-log.md` を読み、スプリント中の進捗を把握する

> **前提ファイルの欠落時**: ファイルが存在しない場合はreadをスキップして実行する。ただし、存在しないファイルがある旨をアラートとして表示する。
> - `planning.md` がない場合: 「⚠️ スプリント計画（planning.md）が見つかりません。完了PBIの情報を直接入力してください」
> - `daily-log.md` がない場合: 「⚠️ デイリーログ（daily-log.md）が見つかりません。スプリント中の経緯の参照はスキップします」

**開始メッセージ:**
```
スプリントレビューの準備を始めましょう！

■ Sprint N レビュー
- スプリントゴール: 「（ゴール）」
- 選定PBI: N件 / XX ポイント

レビューの準備とフィードバック整理、どちらから始めますか？
1. デモ準備（これから実施する場合）
2. フィードバック整理（レビュー実施後の場合）
```

---

### Step 2: デモ準備

#### 2-1. 完了PBIの確認

**問いかけ:**
1. 「完了したPBIを教えてください」
2. 「デモできる状態になっているPBIはどれですか？」
3. 「完了していないが、途中経過を見せたいPBIはありますか？」

**出力:**
```
■ デモ対象PBI:

| # | PBI | 見積もり | 状態 | デモ |
|---|-----|---------|------|------|
| 1 | （タイトル） | X pt | ✅ 完了 | デモあり |
| 2 | （タイトル） | Y pt | ✅ 完了 | デモあり |
| 3 | （タイトル） | Z pt | 🔄 進行中 | 途中経過を共有 |
| 4 | （タイトル） | W pt | 🔲 未着手 | — |
```

#### 2-2. デモシナリオの作成

完了PBIに基づいて、デモの流れを整理する。

**問いかけ:**
1. 「デモの順番はどうしますか？スプリントゴールに最も関連するPBIから見せるのがおすすめです」
2. 「各PBIのデモで、特にPOに見てほしいポイントはありますか？」
3. 「デモ環境は準備できていますか？」

**出力:**
```
■ デモシナリオ:

### デモ1: 「（PBIタイトル）」（X pt）
- 操作手順:
  1. （手順1）
  2. （手順2）
  3. （手順3）
- 見てほしいポイント: （ポイント）
- 想定される質問: （予想される質問とその回答）

### デモ2: 「（PBIタイトル）」（Y pt）
（同様）

### 途中経過の共有: 「（PBIタイトル）」
- 現在の状況: （進捗状況）
- 残りの作業: （残タスク）
- 次スプリントでの完了見込み: （見込み）

■ デモ所要時間（目安）: 約XX分
```

#### 2-3. スプリントゴールとの照合

**出力:**
```
■ スプリントゴール達成状況:

ゴール: 「（ゴール）」
判定: ✅ 達成 / ❌ 未達成 / 🔄 一部達成

（判定の根拠を簡潔に記述）
```

---

### Step 3: フィードバック整理

レビュー実施後に、POやステークホルダーからのフィードバックを整理する。

**問いかけ:**
1. 「レビューで出たフィードバックを教えてください（箇条書きやメモをそのまま貼ってOKです）」
2. 「POからの追加要望や変更依頼はありましたか？」
3. 「ステークホルダーからの質問・懸念はありましたか？」

**フィードバックの分類:**

| 分類 | 説明 | 対応 |
|------|------|------|
| 🆕 新規要望 | 新しい機能やユーザーストーリーの要望 | PBIとしてバックログに追加 |
| 🔄 変更要望 | 既存機能への変更・改善要望 | 既存PBIの更新 or 新PBI作成 |
| 🐛 不具合 | デモ中に発覚した問題 | バグPBIとしてバックログに追加 |
| ❓ 質問 | 確認事項・懸念 | 回答を記録、必要に応じてアクション化 |
| 👍 ポジティブ | 良かった点・評価 | 記録（チームのモチベーション向上に活用） |

**出力:**
```
■ フィードバック整理:

### 🆕 新規要望
- （内容）→ PBI候補: 「（PBIタイトル案）」

### 🔄 変更要望
- （内容）→ 対象PBI: 「（PBIタイトル）」

### 🐛 不具合
- （内容）→ 優先度: 高/中/低

### ❓ 質問・懸念
- （内容）→ 回答: （回答。未回答なら「要確認」）

### 👍 ポジティブフィードバック
- （内容）
```

---

### Step 4: まとめと保存

全体をまとめてユーザーに確認する。

```
■ Sprint N レビュー サマリー

スプリントゴール: 「（ゴール）」→ ✅ 達成 / ❌ 未達成
完了PBI: M / N 件

【デモ実施PBI】
- （PBIタイトル）✅
- （PBIタイトル）✅

【フィードバック】
- 新規要望: X件
- 変更要望: X件
- 不具合: X件

【次スプリントへの持ち越し】
- （PBIタイトル）

この内容でよろしいですか？修正があれば言ってください。
```

ユーザーがOKしたら、`docs/sprints/sprint-N/review.md` に保存する。

---

### Step 5: 完了メッセージ

```
Sprint N のスプリントレビュー記録を保存しました！（docs/sprints/sprint-N/review.md）

■ 完了PBI: M / N 件
■ フィードバック: X件（新規要望 X件、変更要望 X件、不具合 X件）

次のステップ:
- レトロスペクティブを始めましょう（`/retro`）
- フィードバックから生まれた新PBIは、次のプランニングで優先順位づけしましょう
```

---

## 注意事項

- **動くソフトウェアを見せる**: スライドや文書ではなく、実際に動くプロダクトをデモすることが最重要
- **フィードバックを歓迎する姿勢**: 批判ではなく、プロダクトを良くするための貴重な情報として受け止める
- **その場で約束しない**: フィードバックは受け取り、優先順位づけは次のプランニングで行う。「検討します」で受け止める
- **デモ準備とフィードバック整理は別タイミングでもOK**: レビュー前にデモ準備だけ行い、レビュー後にフィードバック整理を行うこともできる
- **PBI状態はDevが管理**: このスキルがPBIの状態を自動更新することはない。フィードバックから生まれたPBIの追加もDevチーム・POが判断する
