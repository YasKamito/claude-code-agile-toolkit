# /refinement ac - 受入条件（AC）品質チェック

受入条件（Acceptance Criteria）の具体性・検証可能性をチェックし、改善案を提示する。
`/refinement check` で T（Testable）に問題があったPBIに対して使うと効果的。

---

## ファイル構成

```
docs/backlog/
├── index.md            # PBIインデックス（対象PBIの選択に使用）
├── PBI-001.md          # 個別PBIファイル（ACを読み取り・更新する）
└── ...
```

---

## 実行フロー

### Step 1: 現在の状態を確認する

以下の情報を収集する:

1. **`docs/project/project-setting.md`** — プロジェクト特性・採用プラクティス
2. **`docs/backlog/index.md`** — PBIインデックス
3. **`docs/backlog/PBI-xxx.md`** — 個別PBIファイル（指定されたPBIのACを読み取る）
4. **要件定義成果物**（`docs/requirements/`）— ビジネスルール・画面定義等（AC改善の根拠として参照）

> **index.md が存在しない場合**: 「⚠️ PBIインデックスが見つかりません。先に `/refinement check` でPBIの詳細化を行うか、チェック対象のACを直接入力してください」と表示する。

**開始メッセージ:**
```
受入条件（AC）品質チェックを始めましょう！

■ 対象: Sprint N の選定PBI（N件）

どのPBIのACをチェックしますか？
- 番号で指定（例: 「1」）
- PBIのタイトルで指定
- ACをそのまま貼り付けてもOKです
```

---

### Step 2: ACの確認

ユーザーからACの情報を受け取る。

**ACが未定義の場合:**
```
このPBIにはまだACが定義されていないようです。
一緒にACを作りましょう！

「このPBIが完了した時、何が確認できればOKですか？」
```

**ACが既にある場合:**
```
以下のACを確認しますね:

1. （AC1の内容）
2. （AC2の内容）
3. （AC3の内容）
```

---

### Step 3: AC品質チェック

各ACについて、以下の3つの観点でチェックする。

| 観点 | 内容 | チェック方法 |
|------|------|-------------|
| 具体性 | 曖昧な表現がないか | 「適切に」「正しく」「速く」等の主観的表現を検出 |
| 検証可能性 | テストで確認できる記述か | 「〇〇した時、△△となる」の形式で書けるか |
| 完了判断 | 完了/未完了を一意に判断できるか | 第三者が見ても同じ判断ができるか |

**チェック結果の提示:**

```
■ AC品質チェック: 「（PBIタイトル）」

### AC1: （ACの内容）
| 観点 | 判定 | コメント |
|------|------|---------|
| 具体性 | ✅ / ⚠️ / ❌ | （コメント） |
| 検証可能性 | ✅ / ⚠️ / ❌ | （コメント） |
| 完了判断 | ✅ / ⚠️ / ❌ | （コメント） |

### AC2: （ACの内容）
（同様）
```

---

### Step 4: 改善案の提示

問題があるACに対して、具体的な改善案を提示する。

**曖昧な表現の改善例:**

| Before（曖昧） | After（具体的） |
|----------------|-----------------|
| 適切に処理される | 〇〇の場合、△△と表示される |
| 正しく動作する | ボタンを押すと、確認ダイアログが表示される |
| 速く表示される | 3秒以内にレスポンスが返る |
| エラーが表示される | 「（具体的なエラーメッセージ）」が赤字で表示される |
| データが保存される | 入力した値がDBに保存され、一覧画面に反映される |
| 適切なバリデーション | 未入力の場合「必須項目です」と表示される |

**改善案の提示フォーマット:**

```
■ AC改善案:

### AC1（改善前）
「適切にエラーが表示される」

### AC1（改善後）
「センサーデータが閾値（設定値）を超えた場合、
 10秒以内にSlack通知が送信され、
 通知メッセージには設備名・センサー種別・検知値・閾値が含まれる」

改善ポイント:
- 「適切に」→ 具体的な条件と結果を明記
- 時間要件（10秒以内）を追加
- 通知内容の具体項目を列挙
```

---

### Step 5: Given-When-Then形式への変換（オプション）

ユーザーが希望する場合、ACをGiven-When-Then形式に変換する。

```
この AC を Given-When-Then 形式に整理しますか？
テストケースとして使いやすくなります。
```

**変換例:**
```
Given: センサーデータの閾値が100に設定されている
When: センサーから値105のデータが受信される
Then: 10秒以内にSlack通知が送信される
And: 通知メッセージに設備名・センサー種別・検知値（105）・閾値（100）が含まれる
```

---

### Step 6: 次のAC or 完了

```
他のACもチェックしますか？
- 同じPBIの別のAC
- 別のPBIのAC
- 「完了」で終了
```

**完了メッセージ:**
```
AC品質チェック完了！

■ チェック結果サマリー:
- チェック済みAC: N件
- 改善提案: M件
- Given-When-Then変換: K件

改善案をPBIに反映してください。
ACが充実すると、スプリント中の手戻りが大幅に減ります！
```

---

## 注意事項

- **ACはチームで合意するもの**: AIが「正解」を出すのではなく、チームが判断するための材料を提供する
- **過剰な詳細化に注意**: ACは「何を検証するか」であって「どう実装するか」ではない。実装の詳細まで書きすぎない
- **ユーザー視点を意識**: 技術的な記述よりも、ユーザーから見た振る舞いで書く
- **Given-When-Then は強制しない**: 形式にこだわるよりも、内容が具体的で検証可能であることが重要
- **project-setting.mdのプロジェクト情報を活用**: プロジェクトのドメインを理解した上で、文脈に合った改善案を提示する
