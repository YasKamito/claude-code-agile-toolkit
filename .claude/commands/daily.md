# /daily - デイリースクラム支援

スプリント中の毎日の同期を支援する。
チームメンバーの進捗を確認し、ブロッカーを早期に検出し、スプリントゴールの達成に向けた軌道修正を対話形式で行う。

結果は `docs/sprints/sprint-N/daily-log.md` に保存する（日付ごとにセクションを追記）。

---

## 利用シナリオ

このスキルは**SM（スクラムマスター）が使用する**ことを想定している。

### パターンA: SMがとりまとめて入力（基本）

1. デイリー前に、各DevメンバーがSlack等に進捗を投稿する
2. SMがその内容をコピーして、このスキルに貼り付ける
3. AIが整理・スプリントゴールとの照合・ブロッカー対応案を提示

### パターンB: MCPでSlackから自動取得（発展）

1. Slack MCPを利用して、デイリー用チャンネルの投稿を自動取得する
2. AIが取得した内容を整理・照合する

> いずれのパターンでも、**Devメンバーが直接このスキルに回答するのではない**。
> SMがファシリテーションとして情報を集約し、AIに整理・分析を依頼する形。

## 実行フロー

### Step 1: 現在の状態を確認する

以下の情報を収集する:

1. このスキルファイルの手順に従って進める（CLAUDE.mdの全文読み込みは不要。プロジェクト情報は project-setting.md から取得する）
2. **`docs/project/project-setting.md`** を読み、プロジェクト情報・採用プラクティス・Playbookを把握する
3. `docs/project/schedule.md` を読み、スプリント日程と会議体情報を把握する
4. `docs/sprints/sprint-N/planning.md` を読み、現在のスプリント情報を把握する
   - スプリントゴール
   - 選定PBI一覧
   - スプリント期間（今日が何日目か — 営業日ベースでカウント）
5. `docs/sprints/sprint-N/daily-log.md` が存在するか確認する
   - **存在する場合**: 前日の記録を読み込み、前日の「今日やること」や未解決のブロッカーを把握する
   - **存在しない場合**: 今スプリント初回のデイリーであることを認識する

> **前提ファイルの欠落時**: ファイルが存在しない場合はreadをスキップして実行する。ただし、存在しないファイルがある旨をアラートとして表示する。
> - `planning.md` がない場合: 「⚠️ スプリント計画（planning.md）が見つかりません。スプリントゴールとPBIの照合はスキップします」
> - `schedule.md` がない場合: 「⚠️ スケジュール（schedule.md）が見つかりません。Day X / Y の表示はスキップします」

**開始メッセージ:**
```
デイリースクラムを始めましょう！

■ Sprint N — Day X / Y（YYYY-MM-DD）
■ スプリントゴール: 「（ゴール）」
■ 選定PBI: N件 / XX ポイント

（前日の未解決ブロッカーがある場合）
⚠️ 前日からの未解決ブロッカー:
- （ブロッカーの内容）

チームメンバーの進捗報告を貼り付けてください。
（Slackの投稿をそのまま貼ってもらってOKです）
```

---

### Step 2: 進捗確認

SMから、各Devメンバーの進捗報告を受け取る。

**問いかけ:**
```
各Devメンバーの進捗報告を教えてください。
以下の3点が含まれていると整理しやすいです:
1. 昨日やったこと
2. 今日やること
3. 困っていること・ブロッカー

Slackの投稿をそのまま貼り付けてもOKです。
```

> **ヒント**: メンバーの報告がまとめて貼られても、個別に貼られてもよい。
> AIが内容を解析してメンバーごとに整理する。

**進捗の記録（メンバーごとにセクション分け）:**
```
■ 進捗報告:

### 田中（BE）
【昨日やったこと】
- （内容）
【今日やること】
- （内容）
【ブロッカー】
- （あれば記載。なければ「なし」）

### 佐藤（BE）
（同様）

### 鈴木（FE）
（同様）
```

---

### Step 3: スプリントゴールとの照合

報告された進捗をスプリントゴールと照らし合わせ、軌道を確認する。

**チェック観点:**

| 観点 | チェック内容 |
|------|-------------|
| ゴール整合 | 今日やることはスプリントゴールの達成に貢献しているか？ |
| PBI進捗 | 選定PBIの消化ペースは順調か？（残り日数に対して） |
| スコープ | スプリント外の作業が入り込んでいないか？ |

> **PBI状態の確認**: PBIの状態（着手中/レビュー中/完了等）はDevメンバーが管理する。
> デイリー前にDevが最新化しておき、デイリーでの決定事項に応じてDevが更新する。
> このスキルがPBIの状態を自動更新することはない。

**警告を出すケース:**
- **ゴールと関係ない作業が多い**: 「今日の作業にスプリントゴールと直接関係ないものが含まれているようです。優先順位は大丈夫ですか？」
- **消化ペースが遅い**: 「スプリント X日目で、選定PBIの進捗が N件/M件 です。ゴール達成に向けてスコープの調整は必要ですか？」
- **スコープクリープの兆候**: 「スプリント計画にないタスクが入っているようです。これはスプリントゴールに必要ですか？それとも次スプリントに回せますか？」

**順調な場合:**
```
✅ スプリントゴールに対して順調です。このペースで進めましょう。
```

---

### Step 4: ブロッカー対応

ブロッカーが報告された場合、対応策を一緒に考える。

**ブロッカーの分類と対応案:**

| 分類 | 例 | 対応案 |
|------|---|--------|
| 技術的 | ライブラリの問題、設計の迷い | スパイク（時間を区切って調査）、ペアプロ |
| 依存 | 他チーム・顧客からの回答待ち | いつまでに必要かを明確にしてエスカレーション |
| 知識 | ドメイン知識不足、仕様の不明点 | POに確認、ドキュメント確認 |
| リソース | メンバー不在、想定外の作業 | スコープ調整、PBIの優先順位変更 |

**出力:**
```
■ ブロッカー対応:
- ⚠️ （ブロッカーの内容）
  → 対応案: （提案）
  → 担当: （誰が対応するか）
  → 期限: （いつまでに解消すべきか）
```

**ブロッカーがない場合はこのステップをスキップする。**

---

### Step 5: まとめと保存

デイリーの内容をまとめてユーザーに確認する。

```
■ デイリーまとめ（YYYY-MM-DD）

Sprint N — Day X / Y
スプリントゴール: 「（ゴール）」

（メンバーごとの進捗サマリー）

【ブロッカー】
- （あれば記載）→ 対応案: （提案）

【軌道確認】
（順調 or 注意点のサマリー）

この内容で記録しますか？
```

ユーザーがOKしたら、`docs/sprints/sprint-N/daily-log.md` に保存する。
- **新規の場合**: ファイルを作成する
- **既存の場合**: 新しい日付のセクションを末尾に追記する

---

### Step 6: 完了メッセージ

```
デイリー完了！（docs/sprints/sprint-N/daily-log.md に記録しました）

■ 今日のフォーカス: （最も重要なタスク1つ）
■ ブロッカー: （あり/なし）

ヒント:
- ブロッカーが解消できない場合はSMに相談しましょう
- スプリント終了時は `/retro` でふりかえりましょう
```

| 項目 | 内容 |
|------|------|

## 注意事項

- **15分以内を目指す**: デイリーは短く。対話が長引きそうなら「この件は別途時間を取って話しましょう」と提案する
- **報告の場ではなく同期の場**: 上司への報告ではなく、チームメンバー同士の情報共有。「チームとして」の視点で進行する
- **ブロッカーの解消はデイリー外で**: デイリー中にブロッカーの解決策を深く議論しない。検出と担当者のアサインまでが目的
- **PBI状態はDevが管理**: デイリー前にDevメンバーがPBIの状態を最新化する。デイリーでの決定事項に応じてDevが更新する。スキルがPBIの状態を自動更新することはない
- **Day X / Y は営業日ベース**: `docs/project/schedule.md` のスプリント開始日・終了日から営業日を算出する。営業日の算出は `schedule.md` の祝日情報を参照する。祝日情報がない場合は `jpholiday` ライブラリで確認する
- **project-setting.mdのPlaybookを参照**: デイリーでの確認事項はPlaybookに定義されている。Playbookに沿って進行する
- **スプリントゴールを常に意識**: 個々のタスクではなく、スプリントゴールの達成に向けた全体の進捗を意識する
