# /retro - レトロスペクティブ支援

スプリント終了時のふりかえりを支援する。
前回の改善アクションのフォロー、スプリント実績の振り返り、リスクの見直し、そして次スプリントへの改善アクションの設定を対話形式で行う。

結果は `docs/sprints/sprint-N/retrospective.md` に保存する。

---

## ファイル構成

```
docs/sprints/sprint-N/
├── planning.md         # /planning の出力
├── daily-log.md        # /daily の出力
└── retrospective.md   # ← このスキルの出力
```

---

## 実行フロー

### Step 1: 現在の状態を確認する

以下の情報を収集する:

1. このスキルファイルの手順に従って進める（CLAUDE.mdの全文読み込みは不要。プロジェクト情報は project-setting.md から取得する）
2. **`docs/project/project-setting.md`** を読み、プロジェクト情報・採用プラクティス・Playbookを把握する
3. `docs/sprints/sprint-N/planning.md` を読み、今スプリントの計画を把握する
   - スプリントゴール
   - 選定PBI一覧・見積もり
   - リスク・依存関係
4. `docs/sprints/sprint-N/daily-log.md` を読み、スプリント中の進捗・ブロッカーの履歴を把握する
5. 前回のレトロを確認する:
   - `docs/sprints/sprint-(N-1)/retrospective.md` が存在するか確認する
   - **存在する場合**: 前回のレトロの改善アクションを読み込む
   - **存在しない場合**: 初回レトロであることを認識する

> **前提ファイルの欠落時**: ファイルが存在しない場合はreadをスキップして実行する。ただし、存在しないファイルがある旨をアラートとして表示する。
> - `planning.md` がない場合: 「⚠️ スプリント計画（planning.md）が見つかりません。計画値との比較はスキップします」
> - `daily-log.md` がない場合: 「⚠️ デイリーログ（daily-log.md）が見つかりません。スプリント中のブロッカー履歴の参照はスキップします」
>
> **スプリント全体の振り返り**: 当スプリントの全記録（planning + daily-log + review）を効率的に要約するには、`sprint-summarizer` サブエージェントを活用する。

**開始メッセージ（初回）:**
```
レトロスペクティブを始めましょう！

初回スプリントのふりかえりですね。
チームがどう感じたかを率直に話せる場にしましょう。
正解・不正解はありません。

■ Sprint 1 の振り返り
- スプリントゴール: 「（ゴール）」
- 期間: YYYY-MM-DD 〜 YYYY-MM-DD
```

**開始メッセージ（2回目以降）:**
```
レトロスペクティブを始めましょう！

■ Sprint N の振り返り
- スプリントゴール: 「（ゴール）」
- 期間: YYYY-MM-DD 〜 YYYY-MM-DD

■ 前回の改善アクション:
- （アクション内容）→ 状況を確認しましょう
```

---

### Step 2: 前回アクションのフォロー

前回のレトロで設定した改善アクションの進捗を確認する。

**問いかけ:**
1. 「前回の改善アクション『（内容）』は実践できましたか？」
2. 「効果はありましたか？」
3. 「継続しますか？それとも完了にしますか？」

**出力:**
```
■ 前回アクションのフォロー:
- （アクション内容）
  → 状態: ✅ 完了 / 🔄 継続 / ❌ 未着手
  → 効果・コメント: （記述）
```

**初回レトロの場合はこのステップをスキップする。**

---

### Step 3: スプリント実績の確認

スプリントの実績を数字で振り返る。

**ヒアリング項目:**
1. 「完了したPBIを教えてください」
2. 「スプリントゴールは達成できましたか？」
3. 「途中で追加・削除したPBIはありますか？」

**出力:**
```
■ スプリント実績:

| 指標 | 計画 | 実績 |
|------|------|------|
| スプリントゴール | 「（ゴール）」 | ✅ 達成 / ❌ 未達成 |
| 完了PBI | N 件 | M 件 |
| ベロシティ | XX pt（計画） | YY pt（実績） |
| スコープ変更 | — | 追加 X件 / 削除 Y件 |

（前スプリントのベロシティがある場合）
→ 前スプリント比: +X pt / -X pt（XX%）
```

> **ベロシティの算出ルール:**
> - **完了したPBIのポイントのみカウント**する。部分完了のPBIはベロシティに含めない
> - 部分完了のPBIは次スプリントに持ち越す
> - 数スプリントにわたって完了できないPBIは**分割を検討**する。分割した上で完了した部分をカウントすることは可能

**ベロシティに対するコメント:**
- **計画超過**: 「計画を上回りました！ただし、次スプリントの計画はこの実績ベースではなく、安定したペースを基準にしましょう」
- **計画未達**: 「計画に届きませんでした。原因を一緒に見ていきましょう（見積もりの精度？想定外の作業？ブロッカー？）」
- **ゴール未達成**: 「ゴールが達成できなかった原因を振り返りましょう。次スプリントで活かせる学びがあるはずです」

---

### Step 4: ふりかえり（Good / Problem / Try）

チームの感想・気づきを引き出す。

**問いかけ:**
1. 「今スプリントで**うまくいったこと**は何ですか？（Good）」
2. 「**うまくいかなかったこと・困ったこと**は何ですか？（Problem）」
3. 「次スプリントで**試してみたいこと**はありますか？（Try）」

> **ファシリテーションのヒント:**
> - 最初にGood（ポジティブな話題）から始めることで、安全な雰囲気を作る
> - Problemでは「誰が悪い」ではなく「何が起きたか」「なぜ起きたか」に焦点を当てる
> - デイリーログからスプリント中のブロッカーや気づきを引用して、「この時こう感じましたか？」と具体的に聞くと話が広がる

**出力:**
```
■ ふりかえり:

【Good — うまくいったこと】
- （内容）

【Problem — うまくいかなかったこと】
- （内容）

【Try — 試してみたいこと】
- （内容）
```

---

### Step 5: リスクバックログの見直し

project-setting.mdの採用プラクティスに「リスクバックログを定期レビュー」が含まれている場合、リスクの見直しを行う。
`docs/project/risk-backlog.md` が存在する場合は読み込み、アクティブなリスクの一覧を提示してレビューする。

**問いかけ:**
1. 「今スプリントで新たに見えてきたリスクはありますか？」
2. 「既存のリスクで解消されたもの、状況が変わったものはありますか？」

**出力（該当がある場合のみ）:**
```
■ リスクバックログ更新（docs/project/risk-backlog.md）:
- 🆕 新規リスク: （内容）→ 対応方針: （提案）
- ✅ 解消済み: （内容）
- 🔄 状況変化: （内容）→ 更新後の対応方針: （提案）
```

リスクバックログの更新がある場合は、`docs/project/risk-backlog.md` にも反映する。

**project-setting.mdの採用プラクティスにリスクバックログが含まれていない場合はこのステップをスキップする。**

---

### Step 6: 改善アクションの設定

ふりかえりの結果から、次スプリントで取り組む改善アクションを設定する。

**ガイドライン:**
- **1つに絞る**: 改善アクションは1つだけにする。複数あると分散して何も変わらない
- **具体的にする**: 「コミュニケーションを良くする」ではなく「PRのレビュー依頼はSlackで即通知する」
- **検証可能にする**: 次のレトロで「やったかどうか」を確認できる形にする

**問いかけ:**
1. 「Tryの中で、次スプリントで一番効果がありそうなのはどれですか？」
2. 「それを具体的なアクションにすると、どうなりますか？」

**曖昧な場合のフォロー:**
- 「次のレトロで『やった/やらなかった』を判断できる表現にしましょう」
- 「誰が・いつ・何をするか、もう少し具体的にできますか？」

**出力:**
```
■ 次スプリントの改善アクション:
「（具体的なアクション）」

検証方法: （次のレトロで何を確認するか）
```

---

### Step 7: レトロ結果の確認と保存

全体をまとめてユーザーに確認する。

```
■ Sprint N レトロスペクティブ サマリー

スプリントゴール: 「（ゴール）」→ ✅ 達成 / ❌ 未達成
ベロシティ: YY pt（計画 XX pt）
完了PBI: M / N 件

前回アクション: ✅ 完了 / 🔄 継続 / ❌ 未着手

【Good】
- （内容）

【Problem】
- （内容）

【Try】
- （内容）

■ 次スプリントの改善アクション:
「（アクション）」

この内容でよろしいですか？修正があれば言ってください。
```

ユーザーがOKしたら、以下を更新する:
1. `docs/sprints/sprint-N/retrospective.md` にレトロ結果を保存する
2. `docs/sprints/sprint-N/planning.md` の「実績」セクションを以下のフィールドで更新する:
   - **ベロシティ**: 完了PBIのポイント合計（部分完了はカウントしない）
   - **完了PBI**: 完了件数 / 計画件数
   - **ゴール達成**: ✅ / ❌
   - **備考**: 学びや気づき（ふりかえりのサマリー）

---

### Step 8: 完了メッセージ

```
Sprint N のレトロスペクティブ完了！（docs/sprints/sprint-N/retrospective.md に保存しました）

■ ベロシティ: YY pt
■ 改善アクション: 「（アクション）」

次のステップ:
- 次スプリントのプランニングを始めましょう（`/planning`）
- 今回のベロシティ（YY pt）を次スプリントの計画に反映します
```

## 出力ファイルのテンプレート

`docs/sprints/sprint-N/retrospective.md` の構造:

```markdown
# レトロスペクティブ — Sprint N

---

## スプリント実績

| 指標 | 計画 | 実績 |
|------|------|------|
| スプリントゴール | 「（ゴール）」 | ✅ / ❌ |
| 完了PBI | N 件 | M 件 |
| ベロシティ | XX pt | YY pt |
| スコープ変更 | — | 追加 X件 / 削除 Y件 |

> ベロシティは完了PBIのポイントのみカウント。部分完了はカウントしない。

## 前回アクションのフォロー
（初回は「初回スプリントのため該当なし」）

## ふりかえり

**Good — うまくいったこと**
- （内容）

**Problem — うまくいかなかったこと**
- （内容）

**Try — 試してみたいこと**
- （内容）

## リスクバックログ更新
- （更新内容。なければ「変更なし」）

## 改善アクション
「（具体的なアクション）」
検証方法: （次のレトロで確認すること）
```

---

## 注意事項

- **安全な場を作る**: レトロは「批判の場」ではなく「学びの場」。誰かを責めるのではなく、プロセスやしくみの改善にフォーカスする
- **改善アクションは1つだけ**: 欲張らない。1つの改善を確実に実行する方が、5つの改善を中途半端にやるより効果的
- **数字と感想の両方を大切にする**: ベロシティだけでなく、チームがどう感じたかも重要な情報
- **前回アクションのフォローを忘れない**: フォローしないと「言いっぱなし」になり、レトロの信頼性が落ちる
- **ベロシティのルール**: 完了PBIのポイントのみカウントする。部分完了はベロシティに含めない。数スプリント残り続けるPBIは分割を検討し、分割後に完了した部分をカウントする
- **スプリント計画の実績も更新する**: レトロで確定したベロシティを planning.md にも反映し、次のプランニングで参照できるようにする
- **project-setting.mdのPlaybookを参照**: レトロで行うべきアクション（リスクバックログレビュー等）はPlaybookに定義されている。Playbookに沿って進行する
