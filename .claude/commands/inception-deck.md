# /inception-deck - インセプションデッキ作成

プロジェクトの方向性をチーム全員で揃えるための「10の問い」に、対話形式で答えていく。
一度に全部やる必要はなく、インクリメンタルに追加・更新できる。

結果は `docs/project/inception-deck.md` に保存する。

---

## 10の問い — 優先度ガイド

### 必須（Sprint 0で必ずやる）

| # | 問い | 目的 |
|---|------|------|
| Q1 | 我々はなぜここにいるのか | チームの存在意義。プロジェクトの「Why」を揃える |
| Q2 | エレベーターピッチ | プロダクトを一言で説明できる状態にする |
| Q3 | やらないことリスト | スコープの境界を明確にする。特にMVPでは超重要 |

### 推奨（早めにやる）

| # | 問い | 目的 |
|---|------|------|
| Q4 | 夜も眠れなくなる問題 | リスクの早期可視化。チームの不安を共有する |
| Q5 | トレードオフスライダー | 品質・スコープ・予算・期間の優先順位を合意する |

### 随時追加

| # | 問い | 目的 |
|---|------|------|
| Q6 | パッケージデザイン | プロダクトの世界観・ユーザーへの見せ方を考える |
| Q7 | ご近所さんを探せ | ステークホルダーや関連チームとの関係を整理する |
| Q8 | 解決案を描く | アーキテクチャや技術選定の全体像を可視化する |
| Q9 | 期間を見極める | リリーススケジュールとマイルストーンを設定する |
| Q10 | 何がどれだけ必要なのか | チーム構成・予算・リソースを明確にする |

## 実行フロー

### Step 1: 現在の状態を確認する

1. このスキルファイルの手順に従って進める（CLAUDE.mdの全文読み込みは不要。プロジェクト情報は project-setting.md から取得する）
2. **`docs/project/project-setting.md`** を読み、プロジェクト情報・採用プラクティス・Playbookを把握する
3. `docs/project/inception-deck.md` が既に存在するか確認する

> **ファイルが存在しない場合**: `docs/project/` ディレクトリが存在しなければ作成する。

**ファイルが存在する場合（追加・更新モード）:**
- 既存の回答を読み込む
- 回答済みの問いと未回答の問いを一覧で示す
- 以下のように表示する:

```
現在のインセプションデッキの状態:

✅ Q1. 我々はなぜここにいるのか
✅ Q2. エレベーターピッチ
✅ Q3. やらないことリスト
🔲 Q4. 夜も眠れなくなる問題
🔲 Q5. トレードオフスライダー
🔲 Q6〜Q10（随時追加）

今日はどの問いに取り組みますか？
- 未回答の問いに答える
- 既存の回答を更新する
- おすすめ: Q4（夜も眠れなくなる問題）— リスクは早めに可視化しておきましょう
```

**ファイルが存在しない場合（初回）:**
- 初回であることを伝え、必須の3問（Q1〜Q3）から始めることを提案する
- ユーザーが「時間がない」と言った場合は、Q1だけでもOKと伝える

```
インセプションデッキを始めましょう！

全部で10の問いがありますが、一度にすべてやる必要はありません。
まずは必須の3問から始めるのがおすすめです:
- Q1. 我々はなぜここにいるのか
- Q2. エレベーターピッチ
- Q3. やらないことリスト

時間がなければQ1だけでも大丈夫です。残りは後から追加できます。
```

---

### Step 2: 対話形式で問いに答える

ユーザーが選んだ問いについて、対話形式で深掘りしていく。

**進め方のルール:**
- 1つの問いにつき、まず問いの意図を簡潔に説明する
- ユーザーの回答を引き出す質問をする（選択肢や具体例を提示）
- 回答が浅い場合は「もう少し聞いていい？」と深掘りする
- ただし完璧を求めない。「今の時点ではこれでOK、後で更新できます」のスタンス
- 1つの問いが終わったら、次の問いに進むか確認する

#### Q1. 我々はなぜここにいるのか

**問いの意図**: このプロジェクトが存在する根本的な理由。顧客のどんな課題を解決するのか。

**ヒアリングの流れ:**
1. 「このプロジェクトが始まった背景を教えてください。お客さんはどんな課題を抱えていますか？」
2. 「その課題を放置するとどうなりますか？（ビジネスインパクト）」
3. 「このプロジェクトが成功したら、お客さんの世界はどう変わりますか？」

**出力フォーマット:**
```markdown
## Q1. 我々はなぜここにいるのか

### 顧客の課題
（課題の記述）

### 放置した場合のインパクト
（ビジネスへの影響）

### プロジェクトが実現する世界
（成功したらどうなるか）
```

#### Q2. エレベーターピッチ

**問いの意図**: プロダクトを30秒で説明できる状態にする。

**ヒアリングの流れ:**
1. 「このプロダクトのメインユーザーは誰ですか？」
2. 「そのユーザーが一番困っていることは？」
3. 「このプロダクトはそれをどう解決しますか？」
4. 「競合や代替手段（Excel管理、目視確認など）と比べて何が違いますか？」

以下のテンプレートに当てはめて一緒に作る:

```
[ターゲットユーザー] が
[解決したい課題] に困っている。
[プロダクト名] は [プロダクトカテゴリ] で、
[主要な価値・メリット] を提供する。
[競合・代替手段] とは違い、
[差別化ポイント] が特徴である。
```

**出力フォーマット:**
```markdown
## Q2. エレベーターピッチ

（テンプレートに当てはめた文章）

### 補足
- **ターゲットユーザー**: （詳細）
- **主要な価値**: （詳細）
- **差別化ポイント**: （詳細）
```

#### Q3. やらないことリスト

**問いの意図**: スコープの境界を明確にする。「やること」より「やらないこと」を決める方が難しく、価値が高い。

**ヒアリングの流れ:**
1. 「このプロダクトに対して、お客さんやチームが期待していそうだけど、今回はやらないことはありますか？」
2. 「将来的にはやるかもしれないけど、MVPでは含めないものは？」
3. 「絶対にやらない（このプロジェクトのスコープ外）と言い切れるものは？」

3つのカテゴリに分類する:

**出力フォーマット:**
```markdown
## Q3. やらないことリスト

| やること（In Scope） | 今回はやらない（Later） | やらない（Out of Scope） |
|---------------------|----------------------|------------------------|
| （項目） | （項目） | （項目） |
```

#### Q4. 夜も眠れなくなる問題

**問いの意図**: チームが抱えるリスクや不安を早期に共有する。

**ヒアリングの流れ:**
1. 「このプロジェクトで一番不安なことは何ですか？」
2. 「技術的に未知の部分はありますか？」
3. 「チームやステークホルダーに関する懸念は？」
4. 「スケジュールや予算で心配なことは？」

**出力フォーマット:**
```markdown
## Q4. 夜も眠れなくなる問題

| リスク | 影響度 | 対応案 |
|--------|--------|--------|
| （リスク） | 高/中/低 | （対応案。なければ「要検討」） |
```

#### Q5. トレードオフスライダー

**問いの意図**: 品質・スコープ・予算・期間のうち、何を優先し何を諦めるかを合意する。

**ヒアリングの流れ:**
1. 以下の4つについて「絶対に譲れない（MAX）」〜「柔軟に対応できる（MIN）」で聞く:
   - スコープ（機能の充実度）
   - 品質（バグの少なさ・安定性）
   - 予算（コスト）
   - 期間（デッドライン）
2. 「MAXにできるのは1つだけ、MINも1つは必要です」と制約を伝える
3. 「なぜその優先順位ですか？」と理由を確認する

**出力フォーマット:**
```markdown
## Q5. トレードオフスライダー

| 項目 | 優先度 | 備考 |
|------|--------|------|
| スコープ | ★★★☆ | （理由） |
| 品質 | ★★☆☆ | （理由） |
| 予算 | ★★★★ | （理由） |
| 期間 | ★☆☆☆ | （理由） |

**MAX**: （項目名）— （理由）
**MIN**: （項目名）— （理由）
```

#### Q6. パッケージデザイン

**問いの意図**: プロダクトの「見た目」や「世界観」をイメージする。ユーザーにどう見えてほしいか。

**ヒアリングの流れ:**
1. 「このプロダクトを一言でキャッチコピーにするとしたら？」
2. 「ユーザーがこのプロダクトを使った時に感じてほしい印象は？（安心感、スピード感、シンプルさなど）」
3. 「競合プロダクトと並んだ時に、何で選んでもらいたいですか？」

**出力フォーマット:**
```markdown
## Q6. パッケージデザイン

- **キャッチコピー**: （一言）
- **ユーザーに感じてほしい印象**: （キーワード）
- **選ばれる理由**: （差別化ポイント）
```

#### Q7. ご近所さんを探せ

**問いの意図**: プロジェクトに影響を与える/受けるステークホルダーや関連システムを把握する。

**ヒアリングの流れ:**
1. 「このプロジェクトの意思決定に関わる人は誰ですか？」
2. 「連携が必要な他チーム・他システムはありますか？」
3. 「このプロジェクトの成果を気にしている人（報告先など）は？」

**出力フォーマット:**
```markdown
## Q7. ご近所さんを探せ

| 名前/チーム | 関わり方 | 重要度 | 備考 |
|------------|---------|--------|------|
| （名前） | （役割・関わり方） | 高/中/低 | （メモ） |
```

#### Q8. 解決案を描く

**問いの意図**: 技術的なアーキテクチャや解決アプローチの全体像を可視化する。

**ヒアリングの流れ:**
1. 「システム全体の構成をざっくり教えてください（どんなコンポーネントがあるか）」
2. 「技術スタックは決まっていますか？制約はありますか？」
3. 「特に検証が必要な技術的チャレンジは？」

**出力フォーマット:**
```markdown
## Q8. 解決案を描く

### アーキテクチャ概要
（テキストベースの構成図 or 説明）

### 技術スタック
| レイヤー | 技術 | 選定理由 |
|---------|------|---------|
| （レイヤー） | （技術名） | （理由） |

### 技術的チャレンジ
- （要検証項目）
```

#### Q9. 期間を見極める

**問いの意図**: リリーススケジュールとマイルストーンを設定する。

**ヒアリングの流れ:**
1. 「いつまでに何ができている必要がありますか？（デッドラインはありますか？）」
2. 「MVPはいつ頃を想定していますか？」
3. 「段階的にリリースする場合、フェーズ分けのイメージはありますか？」

**出力フォーマット:**
```markdown
## Q9. 期間を見極める

| マイルストーン | 目標時期 | 内容 | 備考 |
|--------------|---------|------|------|
| MVP | （時期） | （内容） | （備考） |
| Phase 2 | （時期） | （内容） | （備考） |
```

#### Q10. 何がどれだけ必要なのか

**問いの意図**: チーム構成・予算・必要なリソースを明確にする。

**ヒアリングの流れ:**
1. 「現在のチーム構成で足りていますか？足りないスキルはありますか？」
2. 「予算の制約はありますか？」
3. 「外部リソース（ツール、インフラ、外注など）で必要なものは？」

**出力フォーマット:**
```markdown
## Q10. 何がどれだけ必要なのか

### チーム
| ロール | 人数 | 備考 |
|--------|------|------|
| （ロール） | （人数） | （備考） |

### 予算・リソース
- （必要なリソースのリスト）
```

### Step 3: 回答の確認と調整

各問いの回答を整理して提示し、ユーザーに確認する。

```
Q1の回答をまとめました。内容を確認してください。

（整理された回答）

修正・追記したい点はありますか？問題なければ次の問いに進みます。
```

---

### Step 4: ファイル保存

ユーザーが「今日はここまで」と言ったら、または選択した問いが全て終わったら、`docs/project/inception-deck.md` に保存する。

**ファイルが新規の場合**: テンプレートごと生成する。
**ファイルが既存の場合**: 回答した問いのセクションだけ更新する。未回答の問いのセクションは触らない。

---

### Step 5: 完了メッセージ

**まだ未回答の問いが残っている場合:**
```
インセプションデッキを更新しました！（docs/project/inception-deck.md）

■ 回答済み: Q1, Q2, Q3（3/10）

続きはいつでも `/inception-deck` で追加できます。
```

**全問回答済みの場合:**
```
インセプションデッキ完成です！（docs/project/inception-deck.md）

インセプションデッキは生きたドキュメントです。
状況が変わったら `/inception-deck` で更新しましょう。
```

## 注意事項

- **完璧を求めない**: 「今わかっていること」で十分。後で更新できることを繰り返し伝える
- **プロジェクト特性を参照する**: project-setting.mdのプロジェクト特性（/tailoringの結果）が既にあれば、それを踏まえて質問を調整する（例: 「準委任契約なのでスコープは柔軟ですね」等）
- **チームで使うことを前提に**: 出力はチームメンバーやPOと共有するドキュメント。専門用語を避け、誰が読んでもわかる表現にする
- **既存の回答を壊さない**: 更新モードで他の問いの回答を変更しない。更新対象の問いだけを書き換える
