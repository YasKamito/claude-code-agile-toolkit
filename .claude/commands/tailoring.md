# /tailoring - プロジェクトテーラリング

プロジェクトの特性をヒアリングし、最適なプラクティスセットを推薦する。
結果は `docs/project/project-setting.md` に記録する。

---

## 実行フロー

### Step 1: プロジェクト基本情報

まず、以下の基本情報をユーザーに確認する。

```
プロジェクト名:
顧客:
概要（1-2文で）:
チーム構成（ロール×人数）:
スプリント期間:
開始日（予定）:
```

ユーザーが一部の情報しか持っていない場合は、わかる範囲で埋めて先に進む。

---

### Step 2: プロジェクト特性ヒアリング

`docs/reference/tailoring-criteria.md` を読み込み、C1〜C10 に基づいて、
対話形式で質問していく。

**進め方のルール:**
- 一度に全質問を投げるのではなく、**2-3問ずつ**聞く
- 各質問では選択肢を提示し、ユーザーが選びやすくする
- ユーザーの回答に応じて補足質問をしてもよい（例: 「請負」と答えたら「変更管理の仕組みは決まっていますか？」など）
- ユーザーが迷っている場合は、具体例を出して判断を助ける

**質問順序:**

Round 1（プロジェクトの外部環境）:
- C1. 契約形態
- C2. 顧客の関与度
- C9. 規制・コンプライアンス要件

Round 2（チームの状態）:
- C3. チームのアジャイル成熟度
- C7. チーム規模
- C8. チーム分散度

Round 3（プロジェクトの性質）:
- C4. ドメインの不確実性
- C5. 技術的複雑性
- C6. 品質要求レベル
- C10. プロジェクト期間

各Roundの回答を受け取ったら、「ありがとうございます。次の質問に進みます」と短く返してから次へ。

---

### Step 3: プラクティス推薦

全特性のヒアリングが完了したら、`docs/reference/process-goals.md` を読み込み、
推薦条件に基づいて、このプロジェクトに最適なプラクティスセットを推薦する。

**推薦の出力フォーマット:**

```markdown
## 推薦プラクティスセット

ヒアリング結果に基づき、このプロジェクトに以下のプラクティスを推薦します。

### 立ち上げ（Inception）

| プロセスゴール | 推薦 | 理由 |
|---------------|------|------|
| 1-1. プロダクトの方向性 | インセプションデッキ + カスタマージャーニーマップ | （特性に基づく理由） |
| 1-2. ステークホルダー整理 | ステークホルダーマップ | （理由） |
| ... | ... | ... |

### 計画（Planning）
（同様のフォーマット）

### 実行（Execution）
（同様のフォーマット）

### 品質（Quality）
（同様のフォーマット）

### ふりかえり・改善（Review & Retrospective）
（同様のフォーマット）
```

**推薦時の注意:**
- 各推薦には必ず「なぜこのプロジェクトでこれを選ぶのか」の理由を添える
- 理由はプロジェクト特性（C1〜C10）の回答に紐づけて説明する
- 「デフォルト推薦」のものでも、このプロジェクトの文脈で理由を述べる
- 選択パターン（択一/併用/積み上げ）を尊重する
- 過剰にプラクティスを盛り込まない。プロジェクトに本当に必要なものだけを推薦する

---

### Step 4: ユーザー調整

推薦を提示した後、ユーザーに調整の機会を提供する。

```
この推薦内容について、変更・追加・削除したい項目はありますか？
- 「これは不要」「これを追加したい」「この選択肢を変えたい」など、何でもどうぞ。
- 問題なければ「OK」と言ってください。確定してPlaybookを生成します。
```

ユーザーが変更を求めた場合は、変更を反映して再度推薦を提示する。
変更に対しても「この変更による影響」を補足説明する。

---

### Step 5: project-setting.md更新 & Playbook生成

ユーザーがOKしたら、`docs/project/project-setting.md` を更新する。以下の4セクションを書き換える。

#### 5-1. 「プロジェクト情報」テーブルを記入

Step 1で収集した基本情報を記入する。

#### 5-2. 「プロジェクト特性」セクションを記入

Step 2のヒアリング結果を記録する。フォーマット:

```markdown
## プロジェクト特性（/tailoring ヒアリング結果）

| 特性 | 回答 | 備考 |
|------|------|------|
| C1. 契約形態 | 準委任 | スコープの柔軟性が高い |
| C2. 顧客の関与度 | 密 | POとして深く関与 |
| C3. アジャイル成熟度 | 経験あり | （補足があれば記載） |
| ... | ... | ... |
```

#### 5-3. 「採用プラクティス」セクションを記入

確定したプラクティスセットを記録する。フォーマット:

```markdown
## 採用プラクティス

#### 立ち上げ（Inception）
- **プロダクトの方向性**: インセプションデッキ + カスタマージャーニーマップ
- **ステークホルダー整理**: ステークホルダーマップ
- **初期リスク**: リスクバックログ + プレモーテム
- **働き方**: ワーキングアグリーメント

#### 計画（Planning）
- **リリース計画**: マイルストーンリリース（2スプリントごと）
- ...

（以下、全フェーズ分）
```

#### 5-4. 「Playbook」セクションを生成

採用プラクティスに基づいて、具体的なアクションのPlaybookを生成する。

```markdown
## Playbook

#### Sprint 0（立ち上げスプリント）でやること
- [ ] インセプションデッキ作成（`/inception-deck`）
- [ ] カスタマージャーニーマップ作成
- [ ] ステークホルダーマップ作成
- [ ] リスクバックログ初期化 + プレモーテム実施
- [ ] ワーキングアグリーメント策定
- [ ] DoD定義（コードレビュー必須 + セキュリティレビュー）
- [ ] テスト戦略策定（CI組み込み + ATDD） ⚠️ スキル未実装（手動対応）
- [ ] リリース計画策定（マイルストーン設定） ⚠️ スキル未実装（手動対応）
- [ ] アーキテクチャスパイク（技術検証項目の洗い出し）
- [ ] バックログ初期整備（INVEST基準 + DoR定義）
- [ ] スケジュール策定（`docs/project/schedule.md` にスプリント日程と会議体を記録）

#### 毎スプリントでやること
- [ ] スプリントプランニング（`/planning`）
  - PBI品質確認（`/refinement check` → `/refinement ac`）
  - スプリントゴール設定
- [ ] デイリー（`/daily`）
  - 毎日15分同期
  - ブロッカー検出・対応
- [ ] リスクバックログレビュー（プランニングまたはレトロで）
- [ ] スプリントレビュー（`/sprint-review`）
  - ステークホルダー参加でデモ
- [ ] レトロスペクティブ（`/retro`）
  - 前回アクションのフォロー
  - メトリクス確認（ベロシティ + バーンダウン）

#### マイルストーンごと（2スプリントごと）にやること
- [ ] リリース判定
- [ ] 月次報告作成
- [ ] バックログ全体の優先順位見直し
- [ ] リスクバックログの棚卸し
```

**Playbook生成のルール:**
- 採用プラクティスに含まれないものはPlaybookに入れない
- 対応するskillがある場合はskill名を併記する（`/skill-name`）
- CLAUDE.mdのSkills一覧で 🔲（未実装）のスキルを参照する場合は `⚠️ スキル未実装（手動対応）` を付記する
- チェックリスト形式（`- [ ]`）にして、実際に使えるようにする
- Sprint 0は「一度だけやること」、毎スプリントは「繰り返しやること」、マイルストーンは「定期的にやること」で分ける

---

### Step 6: 完了メッセージ

`docs/project/project-setting.md` 更新完了後、以下のメッセージを表示する:

```
テーラリング完了！docs/project/project-setting.md を更新しました。

■ プロジェクト情報 ✅
■ プロジェクト特性 ✅
■ 採用プラクティス ✅
■ Playbook ✅

次のステップ:
1. Sprint 0のPlaybookに沿って立ち上げを進めましょう
2. まずは `/inception-deck` でプロダクトの方向性を揃えるのがおすすめです
3. `docs/requirements/project-brief.md` にプロジェクトの背景情報を記録しましょう
4. スプリント日程と会議体を `docs/project/schedule.md` に記録しましょう
```

---

## 注意事項

- ヒアリング中にユーザーが「わからない」と答えた場合は、一般的な選択肢を仮置きし、後から変更可能であることを伝える
- テーラリングは一度きりではない。プロジェクトの状況が変わったら `/tailoring` を再実行してプラクティスを見直すことができる
- `docs/project/project-setting.md` を更新する際は、プロジェクト情報・プロジェクト特性・採用プラクティス・Playbookのセクションのみ更新する。CLAUDE.mdは変更しない
