# /requirement - 要件定義支援

プロジェクト特性に応じた要件定義を対話形式で進める。
RDRAの段階的詳細化の考え方をベースに、テーラリング結果に応じてアプローチの深さと手法を調整する。

成果物は `docs/requirements/` に出力する。

---

## トレーサビリティ（ID体系）

要件定義の各成果物はIDで相互に紐づける。RDRAの「関連駆動」の考え方に基づき、
すべてのPBIが「なぜ必要か」を上流の成果物まで辿れるようにする。

```
業務フロー → ストーリー(S/M/A-xxx) → ビジネスルール(BR-xxx)
                                          ↓
                                     画面(SCR-xxx)
                                          ↓
                                     PBI ← 出所を全部記録
                                      ↑
                                     ADR ← 技術判断も紐づく
```

**ID体系:**
- **S-xxx**: 保全担当者等のメインユーザーのストーリー
- **M-xxx**: 管理者・マネージャーのストーリー
- **A-xxx**: システム管理者のストーリー
- **BR-xxx**: ビジネスルール
- **SCR-xxx**: 画面
- **PBI-xxx**: プロダクトバックログアイテム

> IDプレフィックスはプロジェクトのユーザー種別に応じて調整する。
> 重要なのはプレフィックスの文字ではなく、成果物間の紐づきが一貫していること。

---

## アプローチ定義

プロジェクト特性（project-setting.md）に基づき、3段階のアプローチから推奨を決定する。

### Light（軽量）

**推奨条件**: ドメイン不確実性が低い、かつ品質要求が標準、かつ業務が単純（社内ツール、LP、シンプルなWebアプリ等）

**アクティビティ**:
1. ユーザーストーリーマッピング
2. 簡易画面リスト
3. → PBL導出

**成果物**:
- `story-mapping.md`
- `screen-list.md`

---

### Standard（標準）

**推奨条件**: ドメイン不確実性が中、または品質要求が高い、または業務フローの理解が必要（EC、SaaS、一般的な業務アプリ、IoTシステム等）

**アクティビティ**:
1. 業務フロー整理（AS-IS → TO-BE）
2. ユーザーストーリーマッピング
3. 主要ビジネスルール整理
4. 画面遷移・概要
5. 非機能要件スケッチ（主要項目の概要レベル）
6. → PBL導出

**成果物**:
- `business-flow.md`
- `story-mapping.md`
- `business-rules.md`
- `screen-list.md`
- `non-functional.md`（主要項目のみ。Thoroughほど網羅的でなくてよい）

---

### Thorough（網羅的）

**推奨条件**: ドメイン不確実性が高い、かつ（品質要求が高い以上、または規制が厳しい、または複雑な業務ドメイン）（金融、医療、製造業基幹、大規模業務システム等）

RDRAの段階的詳細化をベースに、各段階で使う手法をユーザーと選択する。

**アクティビティ**:
1. システム外部環境の整理（コンテキスト図、アクター、外部システム）
2. 業務フロー分析 — 手法選択:
   - **BPMN / 業務フロー図**: 既存業務の可視化、AS-IS → TO-BE分析に強い
   - **Event Storming**: ドメインイベントからの業務発見、DDDへの接続に強い
   - **併用**: AS-ISをBPMN、TO-BEの発見にEvent Storming
3. 利用シーン — 手法選択:
   - **ユーザーストーリーマッピング**: ユーザー行動フロー中心、MVP定義に強い
   - **ユースケース記述**: アクターとシステムの対話を網羅的に記述、漏れ防止に強い
   - **併用**: ストーリーマッピングで全体像 → 重要部分をユースケースで詳細化
4. ドメインモデル（主要エンティティ、関連、ビジネスルール）
5. 画面・帳票・外部インターフェース
6. 非機能要件（性能、セキュリティ、可用性等）
7. → PBL導出

**成果物**:
- `context.md`（システム外部環境）
- `business-flow.md`
- `story-mapping.md` / `use-cases.md`
- `domain-model.md`
- `screen-list.md`
- `non-functional.md`
- `business-rules.md`

## 実行フロー

### Step 1: コンテキスト収集

#### 1-1. 自動読み込み

以下のファイルが存在すれば読み込む:
- `docs/project/project-setting.md` — テーラリング結果（必須）
- `docs/project/inception-deck.md` — インセプションデッキ
- `docs/requirements/project-brief.md` — プロジェクトブリーフ

#### 1-2. 追加資料の投入

ユーザーに以下を確認する:

```
要件定義に使える資料があれば教えてください。
プロジェクトの理解を深め、より精度の高い要件定義ができます。

例:
- RFP・提案依頼書
- 業務フロー図・業務マニュアル
- 既存システムの画面キャプチャ・仕様書
- プリセールス段階の議事録・ヒアリングメモ
- 業界規制・コンプライアンス文書
- 競合サービスの情報
- ユーザーインタビュー記録

ファイルパス、テキスト貼り付け、どちらでもOKです。
複数ある場合はまとめてどうぞ。なければ「なし」で進めます。
```

投入された資料がある場合:
1. 各資料を読み込む
2. 要件定義に関連する情報を抽出する
3. 以下のフォーマットでサマリーを提示する:

```
■ 資料から読み取れた情報

【業務・ドメイン】
- （業務フローに関する情報）
- （業務ルールに関する情報）

【ユーザー・ステークホルダー】
- （ユーザー種別、利用シーン）

【技術・制約】
- （既存システム、技術制約、規制要件）

【優先度・スコープのヒント】
- （顧客の優先事項、MVP候補）

抜け漏れや認識違いがあれば指摘してください。
```

---

### Step 2: アプローチ提案・合意

Step 1で収集した情報（project-setting.mdの特性 + 追加資料）に基づき、
推奨アプローチを提案する。

**提示フォーマット:**

```
■ 推奨アプローチ: 【Standard】

判断根拠:
- C4（ドメイン不確実性）: 高 → 業務フロー整理が必要
- C6（品質要求）: 標準 → 網羅的なユースケース記述までは不要
- 追加資料から: 既存業務の変革要素あり → AS-IS/TO-BE分析が有効

このアプローチでは以下のアクティビティを行います:
1. 業務フロー整理（AS-IS → TO-BE）
2. ユーザーストーリーマッピング
3. 主要ビジネスルール整理
4. 画面遷移・概要
5. → PBL（プロダクトバックログ）初期版の導出

この内容で進めますか？
変更したい場合（アクティビティの追加/削除、アプローチの変更）は教えてください。
```

**注意:**
- アプローチは厳密な判定ではなく推奨。ユーザーの判断を尊重する
- Thoroughの場合は、各段階の手法選択（BPMN or Event Storming 等）もここで合意する
- 「このプロジェクトならここまでやれば十分」という過不足ない提案を心がける

---

### Step 3: アクティビティ実行

合意したアプローチに応じて、アクティビティを**1つずつ対話的に**進める。

**進め方のルール:**
- 1アクティビティごとに、まず目的と進め方を説明する
- ユーザーとの対話で内容を詰める（一方的に作らない）
- 各アクティビティの成果を確認してから次に進む
- 前のアクティビティの成果を次のインプットとして活用する（段階的詳細化）
- 追加資料から得た情報を積極的に活用し、ユーザーの負担を減らす
- **図表はmermaid記法を積極的に活用する**（業務フロー、画面遷移、データフロー、ナラティブフロー等）
- IDの採番と成果物間の紐づけを一貫して行う（上記「トレーサビリティ」セクション参照）
- アクティビティの過程で**技術的なアーキテクチャ判断が発生した場合は、ADR（Architecture Decision Record）として `docs/adr/` に記録する**ことを提案する

---

#### アクティビティ詳細: 業務フロー整理

**目的**: 現状の業務（AS-IS）とシステム導入後の業務（TO-BE）を可視化し、システムが担う範囲を明確にする。

**進め方**:
1. 追加資料やproject-brief.mdから業務の概要を整理して提示
2. 主要な業務フローを特定（ユーザーと合意）
3. 各フローについてAS-IS → TO-BEを対話で整理
4. システムが介在するポイント（自動化・効率化・新規）を明示

**出力フォーマット（`docs/requirements/business-flow.md`）:**

各フローはmermaid flowchartで図示する。AS-IS（課題箇所を赤系）とTO-BE（システム化ポイントを緑系）で色分けし、視覚的に変化を把握できるようにする。

```markdown
# 業務フロー

## 概要
（業務全体の概要、対象範囲）

## 主要業務フロー

### フロー1: （業務名）

#### AS-IS（現状）

（mermaid flowchart: 課題箇所を fill:#fdd,stroke:#c00 で強調）

**課題:**
- （現状の問題点をリスト）

#### TO-BE（システム導入後）

（mermaid flowchart: システム化ポイントを fill:#dfd,stroke:#0a0 で強調）

#### システムの関与ポイント
| 区分 | 内容 |
|------|------|
| 【自動化】 | （手順X）→ システムが自動実行 |
| 【効率化】 | （手順Y）→ システムが支援、人が判断 |
| 【新規】 | （手順Z）→ 現状にない新しい機能 |

### フロー2: ...
（同様）

## データフロー概要
（mermaid flowchart: システム全体のデータの流れを図示）
```

#### アクティビティ詳細: ユーザーストーリーマッピング

**目的**: ユーザーの行動フローに沿ってストーリーを整理し、リリース単位（MVP、v1、v2...）を定義する。

**進め方**:
1. ユーザー種別（ペルソナ or アクター）を整理
2. 各ユーザーの主要な行動フロー（横軸: ナラティブフロー）を定義
3. 各ステップで必要な機能をストーリーとして洗い出し（縦軸: 詳細度）
4. リリース単位の線引き（MVP / v1 / v2...）をユーザーと合意

**出力フォーマット（`docs/requirements/story-mapping.md`）:**

```markdown
# ユーザーストーリーマッピング

## ユーザー種別
| ユーザー | 説明 | 主な関心事 |
|----------|------|-----------|
| （名前/ロール） | （説明） | （何を達成したいか） |

## ストーリーマップ

### ユーザー: （ユーザー名）

**ナラティブフロー（横軸）:**
（mermaid flowchart LR でナラティブフローを図示）

| アクティビティ | ストーリー | 優先度 | リリース |
|---------------|-----------|--------|---------|
| （アクティビティ1） | ユーザーとして、○○したい。なぜなら△△だから | 高 | MVP |
| | ユーザーとして、○○したい。なぜなら△△だから | 中 | v1 |
| （アクティビティ2） | ... | ... | ... |

## リリース計画（概要）
- **MVP**: （スコープの説明、達成する価値）
- **v1**: （追加スコープ）
- **v2**: （将来スコープ）
```

---

#### アクティビティ詳細: ビジネスルール整理

**目的**: システムが守るべき業務ルール・制約を明文化する。暗黙知を形式知にする。

**進め方**:
1. 業務フローやストーリーマッピングから、判断・計算・制約が含まれるポイントを抽出
2. 各ルールを対話で具体化（条件、例外、境界値）
3. ルールの出所（業界規制、社内ルール、顧客要望等）を記録

**出力フォーマット（`docs/requirements/business-rules.md`）:**

```markdown
# ビジネスルール

## ルール一覧

| ID | カテゴリ | ルール | 条件・詳細 | 出所 | 関連ストーリー |
|----|---------|--------|-----------|------|--------------|
| BR-001 | （分類） | （ルール名） | （具体的な条件・計算式・制約） | （規制/社内/顧客要望） | （関連するストーリーID） |
```

---

#### アクティビティ詳細: 画面遷移・概要

**目的**: ユーザーが触る画面の全体像と遷移を把握する。詳細デザインではなく、機能スコープの確認が目的。

**進め方**:
1. ストーリーマッピングの各ストーリーに必要な画面を洗い出し
2. 画面間の遷移を整理
3. 各画面の主要な要素（表示項目、操作）を概要レベルで記述

**出力フォーマット（`docs/requirements/screen-list.md`）:**

```markdown
# 画面一覧・遷移

## 画面一覧

| ID | 画面名 | 種別 | 概要 | 主要機能 | 関連ストーリー |
|----|--------|------|------|---------|--------------|
| SCR-001 | （画面名） | （一覧/詳細/入力/ダッシュボード等） | （概要） | （主な操作・表示） | （関連ストーリーID） |

## 画面遷移

（mermaid flowchart で画面遷移図を作成。主要導線とサブ導線を subgraph で整理）
（通知（Slack/メール等）からの導線がある場合は別途図示）

## 画面構成イメージ（主要画面）

（ASCII art で主要画面のワイヤーフレームを概要レベルで記述。詳細デザインではなく、機能スコープの確認が目的）
```

### Step 4: 成果物出力

各アクティビティの完了後、成果物を `docs/requirements/` に保存する。

**出力ルール:**
- ファイルが既に存在する場合は上書きではなく更新（既存内容を尊重）
- 各ファイルの末尾に作成日・更新日を記録
- アプローチで不要なファイルは作成しない（Lightなら business-flow.md は作らない）

---

### Step 5: PBL（プロダクトバックログ）初期版の導出

要件定義の成果物からPBLを導出する。

**進め方:**
1. ストーリーマッピングのストーリーをPBIとして整理
2. 業務フロー・ビジネスルールから漏れているPBIがないか確認
3. 非機能要件からのPBI（Standard以上の場合）を追加
4. 優先順位をユーザーと合意（リリース単位と整合させる）
5. 各PBIに概要レベルのAC（受け入れ条件）を付与

**出力フォーマット（`docs/requirements/product-backlog.md`）:**

```markdown
# プロダクトバックログ（初期版）

## バックログ

| ID | ストーリー | AC概要 | SP目安 | リリース | 優先度 | 出所 |
|----|-----------|--------|--------|---------|--------|------|
| PBI-001 | ユーザーとして、○○したい | ・条件1 ・条件2 | S/M/L | MVP | 高 | story-mapping |
| PBI-002 | ... | ... | ... | ... | ... | business-rules |

## 補足
- SP目安はTシャツサイズ（S/M/L/XL）で概算。詳細見積もりはスプリントプランニングで行う
- 出所: どの要件定義成果物から導出されたか（traceability）
- このバックログは初期版。リファインメント（`/refinement check`, `/refinement ac`）で品質を高めていく
```

```
  - 総PBI数: XX件
  - MVP: XX件 / v1: XX件 / v2: XX件

次のステップ:
1. PBIのリファインメント（`/refinement check` → `/refinement ac`）で品質を高めましょう
2. `/planning` でスプリントプランニングに進みましょう
3. 要件定義の内容は開発を進めながら随時更新してください（living document）
```

## 注意事項

- 要件定義は一度で完璧にならない。**イテレーティブに育てる**ことを前提とする
- ユーザーが「わからない」と言った部分は仮置きし、TBDマークをつけて先に進む
- 過度に詳細化しない。**スプリントプランニングで具体化できる粒度**が目安
- 追加資料が多い場合でも、全情報を網羅しようとせず、要件定義に必要な情報に絞って抽出する
- アプローチの途中で「ここはもっと深掘りが必要」「ここは省略してよい」と判断が変わった場合は、柔軟に調整する
