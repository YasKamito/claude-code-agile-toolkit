# /refinement check - PBIリファインメント（INVEST基準）

PBIの詳細化とINVEST基準チェックを対話形式で行う。
粗いPBI（product-backlog.mdのインデックス）から、スプリント投入可能な品質のPBIファイルを作成する。

---

## ファイル構成

```
docs/
├── requirements/
│   ├── product-backlog.md        # /requirement が出力したPBIインデックス（粗い一覧）
│   ├── story-mapping.md          # ストーリーマッピング（出所参照用）
│   ├── business-rules.md         # ビジネスルール（出所参照用）
│   ├── screen-list.md            # 画面一覧（出所参照用）
│   └── business-flow.md          # 業務フロー（出所参照用）
├── backlog/
│   ├── index.md                  # PBIインデックス（ステータス管理付き）
│   ├── PBI-001.md                # 個別PBIファイル
│   ├── PBI-002.md
│   └── ...
└── sprints/sprint-N/
    └── planning.md               # スプリント計画（投入済みPBI）
```

---

## 実行フロー

### Step 1: コンテキスト収集

以下の情報を収集する:

1. **`docs/project/project-setting.md`** — プロジェクト特性・採用プラクティス・Playbook
2. **`docs/backlog/index.md`** — PBIインデックス（存在する場合）
3. **`docs/requirements/product-backlog.md`** — /requirement の出力（初回はここから取り込む）
4. **要件定義成果物** — story-mapping.md、business-rules.md、screen-list.md、business-flow.md（出所参照用）

> **大量PBIの場合**: バックログに20件以上のPBIがある場合は、`backlog-reader` サブエージェントを使って
> ステータス別サマリーを取得し、対象PBIの絞り込みに活用する。
> **要件の文脈抽出**: 複雑なPBIで多くのトレーサビリティ参照がある場合は、`requirements-reader` サブエージェントを使って関連要件のみを抽出する。

#### 初回実行時（index.md が存在しない場合）

product-backlog.md から PBIインデックス（`docs/backlog/index.md`）を生成する:

```
product-backlog.md が見つかりました。PBIインデックスを初期化します。

■ 取り込み対象: MVP XX件 / v1 XX件

docs/backlog/index.md を作成しました。
ここからPBIのリファインメントを進めましょう！
```

**index.md のフォーマット:**

```markdown
# プロダクトバックログ

## MVP

| ID | ストーリー | ステータス | SP | スプリント | 優先度 |
|----|-----------|----------|-----|----------|--------|
| PBI-001 | （タイトル） | Draft / Ready / In Progress / Done | - | - | 高 |
| PBI-002 | （タイトル） | Draft | - | - | 高 |

## v1
（同様）
```

**ステータス定義:**
- **Draft**: 粗い状態。リファインメント未実施
- **Ready**: リファインメント完了。スプリント投入可能
- **In Progress**: スプリントに投入済み
- **Done**: 完了

---

### Step 2: PBI選択

**開始メッセージ:**

```
PBIリファインメントを始めましょう！

■ バックログ状況:
  - Draft（未リファイン）: XX件
  - Ready（投入可能）: XX件
  - In Progress: XX件
  - Done: XX件

■ Draft のPBI一覧:
| # | ID | ストーリー | 優先度 |
|---|-----|-----------|--------|
| 1 | PBI-001 | （タイトル） | 高 |
| 2 | PBI-002 | （タイトル） | 高 |

どのPBIをリファインメントしますか？
- 番号 or IDで指定（例: 「1」「PBI-001」）
- 「優先度高から順に」で自動選択
- 新しいPBIを追加（例: 「新規PBI」）
```

---

### Step 3: PBI詳細化

選択されたPBIについて、要件定義成果物（出所）を自動で参照し、対話的に詳細化する。

#### 3-1. 出所の自動参照

PBIの出所（product-backlog.md に記録されたトレーサビリティ情報）をもとに、関連する要件定義成果物を自動で読み込む。

```
■ PBI-001: 「担当設備の閾値超過をSlackで受け取りたい」

出所を確認しました:
- ストーリー: S-001（story-mapping.md）
- ビジネスルール: BR-001, BR-010, BR-013（business-rules.md）
- 画面: なし（通知のみ）

これらの情報をもとに、受入条件とタスクを整理していきます。
```

#### 3-2. 受入条件（AC）の作成・改善

出所のビジネスルール・画面定義を参照しながら、ACを対話的に作成する。

**ACの品質基準:**

| 観点 | 内容 |
|------|------|
| **具体性** | 曖昧な表現（「適切に」「正しく」「速く」）がないか |
| **検証可能性** | テストで確認できる記述か。「〇〇した時、△△となる」で書けるか |
| **完了判断** | 第三者が見ても完了/未完了を一意に判断できるか |
| **出所との整合** | ビジネスルール・画面定義と矛盾していないか |

**進め方:**
1. 出所（BR/SCR）から導出できるACの叩き台を提示
2. ユーザーと対話で過不足を調整
3. 必要に応じてGiven-When-Then形式に変換

#### 3-3. タスク分解

ACに基づいて、開発タスクの案を提示する。

```
■ タスク案:
- [ ] Slack Webhook設定
- [ ] 通知メッセージテンプレート作成
- [ ] 通知先グループ→Slackチャンネルのマッピング
- [ ] E2Eテスト

過不足や優先順位の調整があれば教えてください。
```

---

### Step 4: INVESTチェック

詳細化されたPBIに対して、INVEST基準の各項目をチェックする。

**INVEST基準:**

| 項目 | 意味 | チェック観点 |
|------|------|-------------|
| **I**ndependent | 独立している | 他のPBIに依存せず単独でデリバリーできるか |
| **N**egotiable | 交渉可能 | HOWを縛りすぎていないか。実装方法の余地があるか |
| **V**aluable | 価値がある | ユーザーやビジネスに価値を提供するか |
| **E**stimable | 見積もり可能 | チームが見積もれる程度に理解できるか |
| **S**mall | 小さい | 1スプリント内で完了できるサイズか |
| **T**estable | テスト可能 | 受入条件が明確で検証できるか |

**チェック結果の提示:**

```
■ INVESTチェック: PBI-001「担当設備の閾値超過をSlackで受け取りたい」

| 項目 | 判定 | コメント |
|------|------|---------|
| I — 独立性 | ✅ / ⚠️ / ❌ | （コメント） |
| N — 交渉可能 | ✅ / ⚠️ / ❌ | （コメント） |
| V — 価値 | ✅ / ⚠️ / ❌ | （コメント） |
| E — 見積もり可能 | ✅ / ⚠️ / ❌ | （コメント） |
| S — 小ささ | ✅ / ⚠️ / ❌ | （コメント） |
| T — テスト可能 | ✅ / ⚠️ / ❌ | （コメント） |

総合: ✅ Ready / ⚠️ 改善推奨 / ❌ 要改善
```

**判定基準:**
- ✅: 問題なし
- ⚠️: 改善の余地あり（そのまま進めることは可能）
- ❌: 要改善（このままでは着手すべきではない）

#### 問題がある場合の対処

| 項目 | よくある問題 | 改善パターン |
|------|-------------|-------------|
| I | 他PBIへの依存 | 依存を解消するか、依存元を先に対応する |
| N | 実装方法が固定 | 「〇〇を使って」→「ユーザーが△△できる」に書き換え |
| V | 技術タスク化 | ユーザーへの価値を明示。価値がなければ統合 |
| E | 不明点が多い | スパイク（技術調査）を先に実施 |
| S | 大きすぎる | 分割を提案（下記パターン） |
| T | ACが曖昧 | Step 3-2に戻ってAC改善 |

**PBI分割パターン（Sに問題がある場合）:**
1. **ワークフロー分割**: ユーザーの操作ステップごと
2. **データバリエーション分割**: 扱うデータの種類ごと
3. **ビジネスルール分割**: ルールの複雑さに応じて
4. **ハッピーパス/エッジケース分割**: 正常系と異常系
5. **CRUD分割**: 作成・参照・更新・削除

---

### Step 5: PBIファイル出力

INVESTチェックが ✅ or ⚠️（許容）になったら、個別PBIファイルを出力する。

**出力先**: `docs/backlog/PBI-xxx.md`

**PBIファイルフォーマット:**

```markdown
# PBI-xxx: （ストーリータイトル）

## 基本情報

| 項目 | 内容 |
|------|------|
| ステータス | Ready |
| 見積もり | （SP or Tシャツサイズ） |
| リリース | MVP / v1 / v2 |
| スプリント | （未割当 or Sprint N） |
| 優先度 | 高 / 中 / 低 |

## ストーリー

（ユーザー）として、（目的）したい。なぜなら（理由）だから。

## 受入条件

- [ ] （AC1: 具体的・検証可能な記述）
- [ ] （AC2）
- [ ] （AC3）

## 出所（トレーサビリティ）

- ストーリー: S-xxx
- ビジネスルール: BR-xxx, BR-xxx
- 画面: SCR-xxx
- ADR: ADR-xxx（関連する場合）

## INVESTチェック結果

| I | N | V | E | S | T | 総合 |
|---|---|---|---|---|---|------|
| ✅ | ✅ | ✅ | ✅ | ✅ | ✅ | ✅ Ready |

## タスク

- [ ] （タスク1）
- [ ] （タスク2）
- [ ] （タスク3）

## メモ

（リファインメント中の議論メモ、懸念事項、決定事項など）
```

**同時にindex.mdを更新:**
- ステータスを Draft → Ready に変更
- SP見積もりを反映

---

### Step 6: 次のPBI or 完了

```
PBI-xxx を Ready にしました！

次のPBIをリファインメントしますか？
- 番号 or IDで指定
- 「完了」で終了
```

全件完了 or ユーザーが終了したら、全体サマリーを表示:

```
■ リファインメント サマリー

今回のセッション:
- リファインメント実施: X件
- Ready化: X件
- 分割: X件 → X件

バックログ全体:
| ステータス | 件数 |
|----------|------|
| Draft | XX件 |
| Ready | XX件 |
| In Progress | XX件 |
| Done | XX件 |

次のステップ:
- Ready のPBIが十分にあれば `/planning` でスプリントプランニングに進めます
- ACの品質をさらに高めたい場合は `/refinement ac` を活用してください
```

---

## 注意事項

- **対話形式で進める**: 一方的にチェック結果を出すのではなく、ユーザーとの対話で情報を引き出しながらチェックする
- **出所の自動参照が肝**: 要件定義成果物をAIが自動で読み込み、文脈を持ったAC作成・改善を行う。これが手動で書くよりも品質が上がるポイント
- **完璧を求めない**: すべてが ✅ である必要はない。⚠️ は「認識した上で進める」判断もあり得る
- **分割の提案は具体的に**: 「分割した方がいい」だけでなく、具体的な分割案（新しいPBIの案）を提示する
- **外部ツールとの連携**: 実プロジェクトではGitHub Projects/Backlog/Jira等と併用する。PBIファイルは外部ツールのチケットと1:1で対応させて運用する
