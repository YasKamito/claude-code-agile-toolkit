# /planning - スプリントプランニング支援

スプリントの目標を設定し、PBIを選定し、チームのキャパシティに基づいて適切な量の計画を立てる。
前スプリントの実績を踏まえて、現実的で達成可能なスプリント計画を対話形式で作成する。

結果は `docs/sprints/sprint-N/planning.md` に保存する。

---

## ファイル構成

```
docs/
├── project/
│   ├── project-setting.md      # プロジェクト全体情報
│   ├── inception-deck.md       # インセプションデッキ
│   └── schedule.md             # スプリント日程・会議体
└── sprints/
    └── sprint-N/
        ├── planning.md         # ← このスキルの出力
        ├── daily-log.md        # /daily の出力
        └── retrospective.md   # /retro の出力
```

---

## 実行フロー

### Step 1: 現在の状態を確認する

以下の情報を収集する:

1. このスキルファイルの手順に従って進める（CLAUDE.mdの全文読み込みは不要。プロジェクト情報は project-setting.md から取得する）
2. **`docs/project/project-setting.md`** を読み、プロジェクト情報・採用プラクティス・Playbookを把握する
3. `docs/project/schedule.md` を読み、スプリント日程と会議体情報を把握する
4. 前スプリントの情報を確認する:
   - `docs/sprints/sprint-(N-1)/planning.md` — 計画と実績
   - `docs/sprints/sprint-(N-1)/retrospective.md` — 改善アクション
   - **これらのファイルが存在しない場合**: 初回スプリントとして扱う。「⚠️ 前スプリントの記録が見つかりません。初回スプリントとして進めます」とアラートを出す

> **前提ファイルの欠落時**: ファイルが存在しない場合はreadをスキップして実行する。ただし、存在しないファイルがある旨を開始メッセージでアラートとして表示する。

> **複数スプリントの実績参照**: 過去のスプリント実績を包括的に把握する必要がある場合は、`sprint-summarizer` サブエージェントを使ってサマリーを取得する。

**初回の場合:**
```
スプリントプランニングを始めましょう！

初回スプリントですね。まずは以下を確認していきます:
1. チームのキャパシティ（今スプリントの稼働状況）
2. スプリントゴールの設定
3. PBIの選定

過去のベロシティがないので、控えめに始めるのがおすすめです。
```

**2回目以降の場合:**
```
Sprint N のプランニングを始めましょう！

■ 前スプリント（Sprint N-1）の実績:
- ベロシティ: X ポイント
- 完了PBI: N 件 / 計画 M 件
- スプリントゴール: [達成 / 未達成]

■ 前回レトロの改善アクション:
- 「（アクション内容）」→ 今スプリントで意識しましょう

この実績を踏まえて計画を立てていきます。
```

---

### Step 2: キャパシティ確認

チームの今スプリントのキャパシティを確認する。

**ヒアリング項目:**
1. 「今スプリントで休暇・不在のメンバーはいますか？」
2. 「スプリント外の作業（会議、他プロジェクト支援など）で時間が減るメンバーは？」
3. 「前スプリントからの持ち越しタスクはありますか？」

**出力:**
```
■ キャパシティ確認:
- フルで稼働できるメンバー: X名
- 稼働減のメンバー: （名前）— 理由（休暇/会議等）
- 持ち越しタスク: あり/なし（ある場合は内容）

→ 前スプリント比で約 XX% のキャパシティです。
```

**キャパシティに基づく警告:**
- 前スプリントよりキャパシティが大幅に減る場合: 「キャパシティが前スプリント比XX%です。PBI量を調整しましょう」
- 持ち越しタスクがある場合: 「持ち越し分を加味して、新規PBIの量を調整しましょう」

---

### Step 3: スプリントゴール設定

**問いかけ:**
1. 「今スプリントで一番達成したいことは何ですか？」
2. 「スプリントが終わった時、何ができていたら成功ですか？」

**ゴール設定のガイドライン:**
- 1つのゴールに絞る（複数ある場合は優先順位をつける）
- 具体的で検証可能な表現にする
- PBI一覧ではなく、ユーザーやビジネスにとっての成果で表現する

**良いゴールの例:**
```
✅ 「設備のセンサーデータを受信し、閾値超過時にSlackに通知が届く状態にする」
❌ 「APIとフロントエンドのタスクを進める」（PBI一覧になっている）
❌ 「できるだけ多くのPBIを消化する」（検証不可能）
```

**曖昧な場合のフォロー:**
- 「それを達成したかどうか、スプリント最終日にどうやって確認できますか？」
- 「そのゴールが達成されると、ユーザーは何ができるようになりますか？」

**出力:**
```
■ スプリントゴール:
「（ゴールの記述）」

検証方法: （どうなったら達成と言えるか）
```

---

### Step 4: PBI選定

ユーザーにPBIの候補を確認し、スプリントに入れるPBIを選定する。

**ヒアリング:**
1. 「バックログの上位にあるPBIを教えてください。または、今スプリントに入れたいPBIはありますか？」
2. ユーザーがPBIを提示したら、以下の観点でチェックする:

**チェック観点:**

| 観点 | チェック内容 |
|------|-------------|
| ゴールとの整合 | スプリントゴールの達成に貢献するPBIか？ |
| サイズ | 1スプリント内で完了できるサイズか？（見積もりがあれば確認） |
| 依存関係 | 他のPBIやチーム外の作業に依存していないか？ |
| 受入条件 | 受入条件（AC）は明確か？（不明確なら `/refinement ac` を推奨） |
| 合計量 | ベロシティに対して量が適切か？ |

**警告を出すケース:**
- **ゴールと無関係なPBI**: 「このPBIはスプリントゴールと直接関係なさそうですが、入れる理由はありますか？」
- **合計がベロシティを超える**: 「選定したPBIの合計が XX ポイントで、前スプリントのベロシティ（YY ポイント）を超えています。どれかを次スプリントに回しませんか？」
- **大きすぎるPBI**: 「このPBIは XX ポイントで、スプリントの半分以上を占めます。分割を検討しませんか？（`/refinement check` で詳細チェックできます）」
- **初回スプリント**: 「初回なのでベロシティの実績がありません。控えめに始めて、次スプリントで調整するのがおすすめです」

**出力:**
```
■ 選定PBI:

| # | PBI | 見積もり | ゴールとの関連 |
|---|-----|---------|--------------|
| 1 | （PBIタイトル） | X pt | ✅ 直接貢献 |
| 2 | （PBIタイトル） | Y pt | ✅ 直接貢献 |
| 3 | （PBIタイトル） | Z pt | ⚠️ 間接的 |

合計: XX ポイント（ベロシティ: YY ポイント → 余裕あり/ちょうど/超過⚠️）
```

---

### Step 5: リスク・依存関係の確認

選定したPBIに対して、リスクや依存関係を確認する。
`docs/project/risk-backlog.md` が存在する場合は読み込み、アクティブなリスクと選定PBIの関連を確認する。

**チェック項目:**
1. 「このスプリントでブロッカーになりそうなことはありますか？」
2. 「チーム外に確認・依頼が必要なことはありますか？」
3. 「技術的に不確実な部分はありますか？（スパイクが必要？）」
4. リスクバックログにアクティブなリスクがある場合: 「リスクバックログに以下のリスクが記録されています。今スプリントに影響はありますか？」

**出力（該当がある場合のみ）:**
```
■ リスク・依存関係:
- ⚠️ （リスクや依存の記述）→ 対応案: （提案）
```

---

### Step 6: スプリント計画の確認と保存

全体をまとめてユーザーに確認する。

```
■ Sprint N 計画サマリー

スプリント期間: YYYY-MM-DD 〜 YYYY-MM-DD
スプリントゴール: 「（ゴール）」

キャパシティ: 前スプリント比 XX%
選定PBI: N 件 / 合計 XX ポイント

| # | PBI | 見積もり |
|---|-----|---------|
| 1 | （PBIタイトル） | X pt |
| ... | ... | ... |

リスク・依存関係:
- （あれば記載）

この内容でよろしいですか？修正があれば言ってください。
```

ユーザーがOKしたら、`docs/sprints/sprint-N/` フォルダを作成し（なければ）、`planning.md` に保存する。

---

### Step 7: 完了メッセージ

```
Sprint N のプランニング完了！（docs/sprints/sprint-N/planning.md に保存しました）

■ スプリントゴール: 「（ゴール）」
■ 選定PBI: N 件 / XX ポイント

次のステップ:
- 毎日のデイリーで進捗を確認しましょう（`/daily`）
- PBIの受入条件に不安があれば `/refinement ac` でチェックできます
- スプリント終了時は `/retro` でふりかえりましょう
```

## 出力ファイルのテンプレート

`docs/sprints/sprint-N/planning.md` の構造:

```markdown
# スプリントプランニング — Sprint N

---

## スプリント情報

| 項目 | 内容 |
|------|------|
| スプリント | Sprint N |
| 期間 | YYYY-MM-DD 〜 YYYY-MM-DD |
| スプリントゴール | 「（ゴール）」 |

## キャパシティ
- フル稼働: X名
- 稼働減: （名前）— 理由
- 持ち越し: あり/なし
- 前スプリント比: XX%（初回スプリントの場合は「初回」）

## 選定PBI

| # | PBI | 見積もり | ゴールとの関連 | 状態 |
|---|-----|---------|--------------|------|
| 1 | （タイトル） | X pt | ✅ | 🔲 |
| 2 | （タイトル） | Y pt | ✅ | 🔲 |

合計: XX ポイント

> PBIの状態はDevメンバーが管理する。デイリー前に最新化し、デイリーでの決定事項に応じて更新する。
> スキルがPBIの状態を自動更新することはない。

## リスク・依存関係
- （あれば記載。なければ「特になし」）

## 実績（スプリント終了後に `/retro` で記入）
- ベロシティ: （完了PBIのポイント合計。部分完了はカウントしない）
- 完了PBI: N / M 件
- ゴール達成: ✅ / ❌
- 備考: （学びや気づき）
```

---

## 注意事項

- **ベロシティは予測ではなく実績ベース**: 「もっとできるはず」ではなく、前スプリントの実績を基準にする
- **初回は控えめに**: ベロシティの実績がない初回スプリントは、感覚の70%程度から始めることを推奨する
- **ゴールを守る**: PBIを詰め込むことより、スプリントゴールの達成を優先する。ゴールに貢献しないPBIは次スプリントに回す勇気を持つ
- **完璧な計画は不要**: スプリントプランニングは「完璧な計画を作る場」ではなく「チームの認識を揃える場」。やりながら調整できることを伝える
- **PBI状態の管理**: PBIの状態（🔲/進行中/完了）はDevメンバーが管理する。このスキルが自動更新することはない
- **部分完了PBIの扱い**: 前スプリントで完了しなかったPBIは持ち越しとして扱う。数スプリント残り続ける場合は分割を検討する
- **project-setting.mdのPlaybookを参照**: プランニング時に行うべきアクション（PBI品質確認等）はPlaybookに定義されている。Playbookに沿って進行する
